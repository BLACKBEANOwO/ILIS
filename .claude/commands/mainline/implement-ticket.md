주어진 이슈 또는 티켓을 구현한다.

인자: $ARGUMENTS (선택. GitLab 이슈 번호, URL, 또는 요구사항 텍스트. 없으면 사용자에게 요구사항을 질문한다.)

전제 조건:
- feature 브랜치에서 작업 중이어야 한다. 아니면 먼저 /mainline:start-feature 실행을 안내한다.

작업 절차:
1. $ARGUMENTS가 이슈 번호나 URL이면 GitLab 이슈를 조회한다 — MCP GitLab 이슈 조회 도구가 있으면 그것을, 없으면 `glab issue view` 명령을 사용한다. 텍스트이면 그대로 요구사항으로 사용한다.
2. 관련 코드, 테스트, 문서를 먼저 찾는다.
3. 요구사항, 가정, 불명확한 점을 코드 변경 단위로 다시 정리한다.
4. 구현에 필요한 정보가 부족하면 먼저 질문하고, 충분하면 진행한다.
5. 플랜 모드로 전환하여 구현 계획을 작성한다. 변경할 파일, 변경 내용, 접근 방식을 정리하여 사용자에게 제시한다.
6. 사용자가 계획을 승인하면 플랜 모드를 종료하고 구현을 시작한다. 수정 요청이 있으면 계획을 수정한다.
7. 현재 작업과 무관한 리팩터링이나 기존 변경사항 되돌리기는 하지 않는다.
8. 최소 변경으로 구현한다.
9. 변경사항에 대한 테스트 코드를 작성한다. 기존 테스트가 있으면 케이스를 추가하고, 없으면 새로 작성한다.
10. 가능한 경우 영향 범위 테스트를 실제로 실행한다. 실행하지 못했으면 이유와 실행 계획을 명시한다.
11. 남은 리스크, 미해결 사항, 후속 작업을 짧게 정리한다.

출력 형식:
- 요구사항 해석 / 가정
- 변경 요약
- 테스트 결과
- 리스크
- 미해결 사항 / 후속 작업
