현재 브랜치의 Merge Request 준비 상태를 점검하고, 대상 브랜치로 rebase한 뒤 GitLab에 MR을 생성한다.

**대상 브랜치** = 팀 규약의 통합 브랜치(기본 `develop`). 이하 절차에서 `<대상>`으로 표기하며, 실제 값으로 치환해 실행한다. 통합 브랜치가 정해지지 않았거나 `develop`이 없으면 사용자에게 확인한 뒤 결정한다.

전제:
- 원격(`origin`)과 GitLab 프로젝트가 있어야 한다. 아직 생성 전이거나 `origin`이 없으면 원격/MR 단계(3~7)를 건너뛰고 그 사실을 알린다.

점검 항목:
1. 변경 목적이 한 문장으로 설명되는지
2. 테스트 결과가 있는지
3. 운영 리스크가 정리됐는지
4. 롤백 방법이 필요한 변경인지

작업 절차:
1. `<대상>` 브랜치를 확정한다(위 규칙). 점검 항목을 확인하고 MR 내용을 작성한다.
2. 현재 브랜치가 작업 브랜치인지 확인한다 — `<대상>`이나 보호 브랜치(`main`/`develop` 등)에 있으면 "작업(feature) 브랜치에서 실행하세요"라고 알리고 중단한다. 필요 시 `/mainline:start-feature`를 안내한다.
3. `git fetch origin`으로 원격 상태를 최신화한다.
4. rebase 전, 이 브랜치가 다른 작업자와 공유 중인지 확인한다(공유 중이면 히스토리 재작성 사전 합의를 받는다). 이어 `git rebase origin/<대상>`을 실행한다.
   - 충돌이 발생하면 충돌 파일 목록을 사용자에게 보고하고 해결을 기다린다. 충돌 내용을 임의로 해결하지 않는다.
   - 사용자가 해결을 마치면 `git rebase --continue`로 진행한다.
   - 사용자가 명시적으로 요청한 경우에만 `git rebase --abort`를 실행한다.
5. rebase 후 로컬 브랜치와 원격 브랜치의 커밋 해시가 달라졌을 수 있으므로 `git push --force-with-lease`로 push한다.
   - `--force-with-lease`가 거부되면(원격에 타인의 커밋이 있음을 의미) 강제 push하지 않고 사용자에게 알린 뒤 중단한다.
   - push할 커밋이 없으면 사용자에게 알린다.
6. GitLab MR을 생성한다 — MCP GitLab MR 생성 도구가 있으면 그것을, 없으면 `glab mr create` 명령을 사용한다(대상 브랜치: `<대상>`).
7. 생성된 MR URL을 출력한다.

주의:
- rebase는 커밋 히스토리를 재작성한다. 이미 다른 작업자와 공유된 feature 브랜치에서는 사전 합의 없이 실행하지 않는다(절차 4에서 확인).

(변경 목적 한 문장 + 주요 변경사항 bullet)

## 테스트
(테스트 결과 또는 테스트 계획)

## 리스크
(운영 리스크 및 영향 범위)

## 롤백 계획
(필요한 경우 롤백 방법)
