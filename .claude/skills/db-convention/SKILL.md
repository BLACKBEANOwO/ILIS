# DB 규약 (PostgreSQL)

ILIS 데이터베이스의 설계·DDL·SQL 규약.

> 스택: **PostgreSQL 17.x** · psql 17.x · DBeaver
> DBMS는 **작업 전제**다 — 정식 확정은 출장 담당자 협의 결과를 따른다.
> 근거: 「3.4 DataBase 개발가이드(PostgreSQL) v0.50」·「명명규칙 기본안 v0.50」
> 상위 규약: `backend/CLAUDE.md` (kit 에서는 `CLAUDE.backend.md`)

## When to Use

- 테이블·컬럼·인덱스 설계, DDL 작성
- ERD 작성·검토
- 마이그레이션 스크립트 작성
- MyBatis XML의 SQL 작성·리뷰

## 명명 규칙

### 테이블·컬럼 — snake_case 의미 기반

```sql
CREATE TABLE law_article (
    law_article_id  BIGINT,
    law_id          BIGINT,
    article_no      VARCHAR(20),
    created_at      TIMESTAMP
);
```

- 테이블명은 **단수형** — `law` (O) / `laws` (X)
- 컬럼명은 의미를 담는다. 축약하지 않는다 — `customer_count` (O) / `cust_cnt` (X)
- 도메인 용어는 원어를 유지한다 — 인니 법령의 `pasal`·`ayat`·`bab`
- PK 컬럼은 `<테이블명>_id` — `law.law_id`, `law_article.law_article_id`
- FK 컬럼은 참조 대상의 PK 컬럼명을 그대로 — `law_article.law_id`
- boolean 컬럼은 `is_`·`has_` 접두 — `is_deleted`, `has_attachment`

> **사내 DB가이드의 코드형 명명(`TBB005`·`EMPNO`)을 쓰지 않는다.**
> 해당 절은 `doc_id: NFMI-ST-FP-034` 이고 프로젝트 약자를 `NFMI`(의료보험 영문 약자)로 명시한다.
> 또한 코드형은 응용업무·단위업무 시스템 분류코드 체계를 전제하는데,
> **현재까지 작성된 ILIS 산출물에는 그 분류 체계가 정의되어 있지 않다.**
> 「명명규칙 기본안 v0.50」이 "코드 작성과 유지보수 중심, 협업 프로젝트"에 의미 기반을 추천한다.

### 제약·인덱스 — 접두사

DB가이드의 자원 구분 체계를 접두사 형태로 채택한다. 오류 메시지에서 어떤 제약인지 바로 보인다.

| 대상 | 형식 | 예시 |
|---|---|---|
| Primary Key | `pk_<테이블>` | `pk_law` |
| Foreign Key | `fk_<테이블>_<참조테이블>` | `fk_law_article_law` |
| Unique | `uk_<테이블>_<컬럼>` | `uk_law_law_no` |
| Index | `ix_<테이블>_<컬럼>` | `ix_law_status` |
| Sequence (명시 생성분) | `sq_<의미>` | `sq_law_no` |
| View | `vw_<의미>` | `vw_law_current` |

복합 컬럼은 `_` 로 잇는다 — `ix_law_article_law_id_article_no`.
63자를 넘으면 PostgreSQL이 잘라내므로 그때는 의미를 줄여 짓는다.

> **시퀀스 예외** — PK를 `IDENTITY`로 만들면 PostgreSQL이 시퀀스를 자동 생성하고
> 이름도 `<테이블>_<컬럼>_seq` 로 정한다. **이건 그대로 둔다.**
> `sq_` 규칙은 `CREATE SEQUENCE` 로 직접 만드는 채번용 시퀀스에만 적용한다.

### 테이블스페이스·데이터 디렉토리

| 대상 | 형식 | 예시 |
|---|---|---|
| Tablespace | `ilis_<업무>_<용도>` | `ilis_law_data`, `ilis_law_indx` |
| Data Directory | `/data/pg_node/<tablespace>_data` | 루트는 서버 환경에 따름 |

> 테이블스페이스는 **DBA가 사전 구성한 환경에서만** 사용한다. DDL에 기본으로 넣지 않는다(§DDL 작성 기준).

## 컬럼 순서 — 4단계

의미 없는 순서로 나열하지 않는다. 아래 순위대로 배치한다.

| 순위 | 대상 |
|---|---|
| **1** | PRIMARY KEY 컬럼. 복합키면 `=`·`LIKE` 비교에 자주 쓰는 것을 앞에 |
| **2** | 입력이 필수(MANDATORY)인 컬럼. UNIQUE·후보키, 테이블에서 의미가 큰 것 우선 |
| **3** | FOREIGN KEY, 그리고 상태(STATUS) 성격의 컬럼 |
| **4** | 선택(OPTIONAL) 컬럼. 입력 시점이 같은 것끼리 인접, NULL 빈도가 낮은 순서로 |

공통 감사 컬럼(`created_at` 등)은 4순위 뒤에 모아 둔다.

```sql
CREATE TABLE law (
    law_id       BIGINT       NOT NULL,   -- ① PK
    law_no       VARCHAR(50)  NOT NULL,   -- ② 필수 + UNIQUE
    title        VARCHAR(500) NOT NULL,   -- ② 필수, 의미 큼
    lang_cd      CHAR(2)      NOT NULL,   -- ② 필수
    status       VARCHAR(20)  NOT NULL,   -- ③ 상태
    agency_id    BIGINT,                  -- ③ FK
    promulgated_on DATE,                  -- ④ 선택, NULL 적음
    note         TEXT,                    -- ④ 선택, NULL 많음
    created_at   TIMESTAMP    NOT NULL,   -- 감사 컬럼
    created_by   VARCHAR(50)  NOT NULL,
    updated_at   TIMESTAMP,
    updated_by   VARCHAR(50)
);
```

## 데이터 타입 선정

> ⚠️ 사내 DB가이드에는 Oracle 잔재가 남아 있다 — `NUMBER` 타입과 "DATE는 7바이트 고정" 설명.
> PostgreSQL에 `NUMBER`는 없고(`NUMERIC`), `DATE`는 4바이트다. 아래가 정정본이다.

### 문자

| 타입 | 대상 | 길이 |
|---|---|---|
| `CHAR(n)` | 길이가 **고정이고 변하지 않는** 코드·구분값 | 정확한 길이 |
| `VARCHAR(n)` | 그 외 대부분. `LIKE` 비교 대상 컬럼 | 최대 길이 (모르면 예상치 +10%) |
| `TEXT` | 길이 상한이 의미 없는 본문 | — |

**코드 컬럼도 자릿수가 바뀔 가능성이 있으면 `VARCHAR`를 쓴다.** 업무 변경으로 코드 자리수가 늘거나
NULL이 들어가면 `CHAR`는 공백 패딩 때문에 비교가 어긋난다. 법령 분류 코드가 여기 해당한다.

날짜를 문자로 저장할 때는 `CHAR(8)`에 `YYYYMMDD`. 다만 **날짜 연산이 필요하면 `DATE`를 쓴다.**

### 숫자

| 타입 | 대상 |
|---|---|
| `BIGINT` | PK·FK, 일련번호 |
| `INTEGER` | 개수·순번 등 범위가 확실한 정수 |
| `NUMERIC(p, s)` | 금액·비율 등 **정확도가 중요한** 소수 |
| `DOUBLE PRECISION` | 통계·과학 계산 (금액에 쓰지 않는다) |

PK는 `BIGINT GENERATED BY DEFAULT AS IDENTITY` 를 쓴다 (`SERIAL`은 레거시).

### 날짜·시각

| 타입 | 대상 | 크기 |
|---|---|---|
| `DATE` | 날짜만 (공포일·시행일·선고일) | 4바이트 |
| `TIMESTAMP` | 날짜+시각, 시간대 없음 | 8바이트 |
| `TIMESTAMPTZ` | 시간대가 의미 있는 시각 | 8바이트 |

감사 컬럼(`created_at`)은 `TIMESTAMP`. 시간대 정책(WIB 등)은 미확정이므로
확정 전까지 `TIMESTAMPTZ`를 새로 도입하지 않는다.

### 기타

- boolean → `BOOLEAN` (`CHAR(1)`에 `'Y'/'N'` 을 쓰지 않는다)
- JSON → `JSONB` (`JSON` 아님 — 인덱싱·연산 지원)
- 파일은 DB에 넣지 않는다. **경로만 저장**한다 (BLOB 미사용 — 기 확정 사항)

## 공통 감사 컬럼

모든 업무 테이블에 둔다. 코드 테이블·매핑 테이블도 예외가 아니다.

```sql
created_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
created_by  VARCHAR(50)  NOT NULL,
updated_at  TIMESTAMP,
updated_by  VARCHAR(50)
```

`created_by`·`updated_by`는 **애플리케이션이 채운다.** 트리거로 채우지 않는다(§금지).

삭제는 물리 삭제를 기본으로 하되, 이력이 필요한 테이블만 `is_deleted BOOLEAN NOT NULL DEFAULT FALSE`
를 두고 논리 삭제한다. **한 테이블에서 두 방식을 섞지 않는다.**

## 마이그레이션 (Flyway)

DDL을 psql로 직접 실행하지 않는다. **백엔드 앱이 기동할 때 Flyway가 자동으로 적용**한다.

```
backend 기동
   ↓
Flyway 가 flyway_schema_history 테이블을 읽는다   ← 어디까지 적용했는지 기록
   ↓
아직 적용 안 된 V*.sql 만 번호 순서대로 실행       ← resources/db/migration/
   ↓
스키마 반영 완료 → 앱 기동 계속
```

### 파일 명명

`V<번호>__<설명>.sql` — 번호 뒤 **밑줄 2개**다.

```
V1__create_law_schema.sql
V2__create_review_schema.sql
V3__add_law_note_column.sql
```

### 적용된 파일은 수정하지 않는다

Flyway는 각 파일의 **체크섬을 기록**한다. 이미 적용된 파일의 내용을 고치면 다음 기동 때
검증에 실패해 **앱이 아예 뜨지 않는다.**

- 스키마를 바꿔야 하면 **새 파일**을 추가한다 (`V4__add_law_note_column.sql`)
- **아직 어느 환경에도 적용되지 않은 파일만** 수정할 수 있다
- 로컬에서만 적용한 파일을 고쳤다면 로컬 DB를 지우고 다시 올린다
- `flyway.clean`은 운영에서 비활성화한다 (전체 스키마 삭제)

### 마이그레이션 단위 — 논리 변경 하나 = 파일 하나

**하나의 논리적 변경을 여러 버전으로 쪼개지 않는다.** 테이블·제약·인덱스를 V1/V2/V3로 나누면
중간 버전이 "테이블은 있는데 PK가 없는" 상태로 남고, 그 시점으로 되돌아갈 수 없다.
PostgreSQL은 DDL도 트랜잭션으로 처리하므로 한 파일에 담아도 원자적으로 적용된다.

```
resources/db/migration/
├── V1__create_law_schema.sql      # 초기 스키마 — 테이블+제약+인덱스 한 벌
├── V2__create_review_schema.sql   # 검수 도메인 한 벌
└── V3__add_law_note_column.sql    # 컬럼 추가 하나
```

**초기 스키마처럼 규모가 큰 경우만** 도메인 단위로 나눈다(법령·판례·검수·배정).
그 안에서 테이블·제약·인덱스는 한 파일에 함께 둔다.

파일 안에서는 순서를 지킨다 — 테이블 → 제약(`ALTER TABLE ADD`) → 인덱스.

### 1) 테이블

제약 중 `NOT NULL`·`CHECK`·`DEFAULT`만 인라인으로 둔다. 이름은 부여하지 않아도 된다.

```sql
CREATE TABLE law (
    law_id     BIGINT       NOT NULL GENERATED BY DEFAULT AS IDENTITY,
    law_no     VARCHAR(50)  NOT NULL,
    title      VARCHAR(500) NOT NULL,
    status     VARCHAR(20)  NOT NULL DEFAULT 'DRAFT'
               CHECK (status IN ('DRAFT','REVIEW_WAIT','REVIEW_DONE','SERVICE')),
    created_at TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_by VARCHAR(50)  NOT NULL
) WITH (
    fillfactor = 70          -- 갱신이 잦다고 확인된 테이블에만
);
```

> **`TABLESPACE` 절은 기본 DDL에 넣지 않는다.** 해당 테이블스페이스가 환경에 없거나
> 앱 계정에 사용 권한이 없으면 **Flyway가 즉시 실패해 앱이 뜨지 않는다.**
> DBA가 테이블스페이스를 사전 구성한 환경에서만, 그 사실을 확인한 뒤 붙인다.

### 2) 제약 — `ALTER TABLE ADD`

PK·FK·UNIQUE는 이름을 반드시 부여한다. 테이블 생성 뒤에 이어서 쓴다.

```sql
ALTER TABLE law ADD CONSTRAINT pk_law PRIMARY KEY (law_id);

ALTER TABLE law ADD CONSTRAINT uk_law_law_no UNIQUE (law_no);

ALTER TABLE law_article ADD CONSTRAINT fk_law_article_law
    FOREIGN KEY (law_id) REFERENCES law (law_id);
```

FK에 `ON DELETE CASCADE`를 기본으로 쓰지 않는다. 연쇄 삭제는 애플리케이션이 통제한다.

### 3) 인덱스

```sql
CREATE INDEX ix_law_status ON law (status)
    WITH (fillfactor = 80);

-- 부분 인덱스 — 논리 삭제 테이블에서 유효 행만
CREATE INDEX ix_law_article_law_id ON law_article (law_id)
    WHERE is_deleted = FALSE;
```

FK 컬럼에는 인덱스를 만든다 — PostgreSQL은 FK에 인덱스를 자동 생성하지 않는다.

## STORAGE PARAMETER

| 옵션 | 용도 | 권장 |
|---|---|---|
| `fillfactor` | 페이지당 채움 비율 | 갱신 잦은 테이블 70, 조회 위주 100(기본), 인덱스 80 |
| `autovacuum_enabled` | 자동 VACUUM | **DDL에 쓰지 않는다** (기본값 `true`) |
| `toast_tuple_target` | TOAST 분리 임계 | **기본값 유지** (아래 참조) |
| `parallel_workers` | 병렬 스캔 워커 수 | 대용량 테이블에만 명시 |

**`toast_tuple_target`을 임의로 바꾸지 않는다.** "큰 텍스트가 있으니 올린다"는 잘못된 직관이다 —
이 값을 **올리면** 더 많은 데이터가 heap 에 남고, **내리면** 더 일찍 TOAST 로 분리된다.
압축·inline 저장 전략에 함께 영향을 주므로 실측 없이 조정하면 역효과가 난다.
기본값(2048)을 두고, 성능 문제가 측정된 뒤 DBA와 함께 조정한다.

`fillfactor` 도 마찬가지로 갱신이 잦다고 확인된 테이블에만 낮춘다.

**기본값과 같은 값을 DDL에 명시하지 않는다.** 의미 없이 길어지고, 나중에 PostgreSQL 기본값이
바뀌었을 때 의도적으로 고정한 것인지 그냥 적어둔 것인지 구분되지 않는다.

## SQL 작성 기준

- **키워드는 대문자**, 식별자(테이블·컬럼)는 소문자
- 조인은 **`JOIN ... ON`** 으로 명시한다
- Alias는 **의미가 드러나게** 짓고(`law`·`article`) 모든 컬럼에 접두를 붙인다
- 절 순서: `SELECT` → `FROM` → `JOIN` → `WHERE` → `GROUP BY` → `HAVING` → `ORDER BY`
- `SELECT *` 금지 — 컬럼을 명시한다

```sql
SELECT law.law_id, law.law_no, law.title, article.article_no
  FROM law
  JOIN law_article article ON article.law_id = law.law_id
 WHERE law.title LIKE #{keyword} || '%'
   AND law.promulgated_on BETWEEN #{fromDate} AND #{toDate}
   AND law.status = #{status}
 ORDER BY law.law_id;
```

> **사내 DB가이드의 SQL 작성 기준 2개를 채택하지 않았다.**
> 콤마 조인(`FROM law A, law_article B WHERE A.x = B.x`)은 조인 조건을 빠뜨리면
> **경고 없이 카티션 곱**이 되고, `A`·`B`·`C` 순번 Alias는 쿼리가 길어질수록 무엇을 가리키는지 알 수 없다.
> 나머지 기준(키워드 대문자, 절 순서, `SELECT *` 금지)은 그대로 따른다.

MyBatis에서는 `#{}` 를 쓴다. `${}` 는 화이트리스트를 통과한 값(정렬 컬럼·방향)에만 (`backend/CLAUDE.md`).

## 금지

| 대상 | 사유 |
|---|---|
| **트리거** | 로직이 DB에 숨어 추적이 안 된다. 감사 컬럼도 애플리케이션이 채운다 |
| **저장 프로시저 / 함수 기반 업무 로직** | 업무 로직은 Service 계층에 둔다 |
| **DB Link** | 시스템 간 연계는 API로 한다 |
| `SELECT *` | 컬럼 추가 시 예기치 않은 동작 |
| 물리 파일 저장(BLOB) | 파일은 파일시스템, DB는 경로만 (기 확정) |
| 코드값을 아무 데나 나열 | 아래 §코드값 판단 기준 참조 |

---

## 코드값 — `CHECK` 인가 코드 테이블인가

둘 다 쓴다. **값이 늘 때 애플리케이션도 바뀌어야 하는가**로 가른다.

| 성격 | 방식 | 예시 |
|---|---|---|
| **상태·구분값** — 값이 늘면 화면·로직도 바뀐다 | 컬럼 + `CHECK` | `law.status`, `review_request.status` |
| **업무 분류 코드** — 운영 중 관리자가 추가한다 | 코드 테이블 + FK | 법령 종류, 소관 기관, 언어 |

**상태값을 코드 테이블로 빼지 않는다.** 상태가 하나 늘면 어차피 전이 규칙·화면·권한이 함께 바뀌므로
마이그레이션이 같이 나가는 게 맞고, `CHECK`가 있으면 잘못된 값이 DB에 들어오지 않는다.

```sql
-- 상태 = CHECK
status VARCHAR(20) NOT NULL DEFAULT 'DRAFT'
       CHECK (status IN ('DRAFT','REVIEW_WAIT','REVIEW_DONE','SERVICE'))

-- 업무 분류 = 코드 테이블 + FK
law_type_cd VARCHAR(20) NOT NULL      -- REFERENCES code_law_type(law_type_cd)
```

**반대로 업무 분류 코드를 `CHECK`로 나열하지 않는다.** 관리자가 값을 추가할 때마다
마이그레이션을 배포해야 한다.

## 감사 컬럼과 업무 행위자 컬럼

`created_by`는 **행을 만든 계정**이다. 업무상 행위자(요청자·작성자·검수자)와 같으면 감사 컬럼으로 갈음하고
별도 컬럼을 만들지 않는다 — 항상 같은 값이 들어가는 중복이 된다.

**다를 수 있으면 별도 컬럼을 둔다.** 대리 등록(담당자가 다른 사람 몫을 대신 입력),
이관(담당자가 바뀜) 같은 업무가 있으면 `requested_by`와 `created_by`가 갈린다.

> ILIS는 배정 기반 단계 진행이라 대체로 **행 생성자 = 업무 행위자**다.
> 도메인별로 예외가 있는지는 기능명세서를 확인하고 정한다.

## 백엔드 규약과의 정합

- DB는 `snake_case`, 자바는 `camelCase` — MyBatis `map-underscore-to-camel-case: true` 로 자동 매핑
- 매핑이 안 되는 컬럼만 `<resultMap>` 에 명시
- `LocalDate` ↔ `DATE` · `LocalDateTime` ↔ `TIMESTAMP`
- `OffsetDateTime` ↔ `TIMESTAMPTZ` — **시간대 정책 확정 후 도입**한다.
  그전까지는 백엔드도 `OffsetDateTime` 을 새로 쓰지 않는다(`backend/CLAUDE.md` §날짜·시각 표기)
- 스키마 변경은 Flyway 마이그레이션으로만 (§마이그레이션). 애플리케이션 코드가 DDL을 실행하지 않는다
