# 기능명세서 산출물 포맷

> **이 문서의 목적**: Claude가 기능명세서를 생성할 때 따라야 하는 **산출물 구조와 작성 규칙**을 정의합니다.
> 작성자가 직접 수정할 필요는 없으며, Claude가 자동으로 참조합니다.

> **📌 공통 원칙 — 문서 참조**
> 이 문서의 기술 스택 섹션 또는 본 스킬 전체 진행 중 **외부 라이브러리/프레임워크(React, React Hook Form, Zod, Spring Boot, Spring Security, MyBatis, PostgreSQL 등)의 문서·API·설정값·버전별 동작**을 확인해야 할 때는 반드시 **`use context7`** 로 최신 공식 문서를 조회한 뒤 명세서에 반영한다. 훈련 데이터의 구버전 정보로 작성하지 않는다.

> **📌 섹션 번호 정의 (정본)**
> 이 문서에서 `§N` 은 **`docs/template.md` 의 질문 섹션 번호(§1~§10)** 를 가리킨다. 정본은 다음과 같다.
> §1 기능 개요 · §2 데이터 모델 · §3 화면 레이아웃 · §4 기능 상세 · §5 API 스펙 · §6 비즈니스 규칙 · §7 인증·보안 · §8 기술 스택 · §9 참고자료 · §10 코드 품질 스킬.
> SKILL.md · template.md · example.md 와 이 문서는 모두 이 번호 체계를 공유한다. 아래 "산출물 구조" 의 항목은 이 §번호로 매핑하여 표기한다.

## 스코프별 최소 필수 필드

기능명세서는 §1에서 선언한 **스코프**에 따라 필수 필드가 달라진다. 스코프와 무관한 섹션은 명세서에서 `"해당 없음 (스코프: X)"` 로 표기한다.

| 스코프 | 필수 필드 |
|---|---|
| `full` | 메타 헤더 · §2 데이터 모델 · §3 화면 레이아웃(3-1, 3-5 권장, 3-6 **필수**) · §4 기능 상세 · §5 API 스펙(5-1, 5-3, 5-4 **필수**) · §6 비즈니스 규칙 · §7 인증/보안 |
| `frontend-only` | 메타 헤더 · §3(3-1 **필수**, 3-5, 3-6 **필수**) · §4 기능 상세 |
| `frontend-with-mock` | 메타 헤더 · §3(3-1, 3-5, 3-6 **필수**) · §4 · §5-1(API 목록) · 5-2(응답 스키마 힌트, mock 용) |
| `backend-only` | 메타 헤더 · §2 · §5(5-1, 5-3, 5-4 **필수**) · §6 · §7 |

**모든 스코프 공통**: 문서 메타 헤더의 `스코프` 항목은 필수. 명세서 최상단 표에 스코프 값을 기재한다.

## 파일 경로 규약

- **기능명세서 저장 위치**: `docs/<현재브랜치>/<기능명>.md`
  - `<현재브랜치>`는 `git branch --show-current` 결과를 그대로 사용한다 (슬래시 포함 시 중첩 폴더).
  - 폴더가 없으면 `mkdir -p`로 생성한다.
- `<기능명>` 은 **소문자 + 언더스코어(`_`)** 로 구성한다. (공백/한글/하이픈 금지)
- 예 (브랜치가 `feature/agent-studio`인 경우): `docs/feature/agent-studio/calc_product_basic.md`, `docs/feature/agent-studio/premium_calculation.md`
- 동일 파일명이 이미 존재하면 덮어쓰지 않고 사용자에게 처리 방법을 확인한다.
- 스킬 내부 리소스는 `.claude/skills/feature-spec/` 에, 템플릿은 `docs/template.md` 에 위치한다 (브랜치 무관 영구 문서).

## 문서 메타 헤더 (필수)

모든 기능명세서는 맨 앞에 **다음 표**를 포함한다. 해당 없는 항목은 `-` 로 표기한다.

```markdown
# {기능명 한국어} 기능명세서

| 항목 | 값 |
|---|---|
| 기능명 | (예: 상품기초정보) |
| 스코프 | (예: full / frontend-only / frontend-with-mock / backend-only) |
| 작성일 | YYYY-MM-DD |
| 상태 | draft / 검토중 / 확정 |
| 관련 테이블 | (예: product, tree_node) — 스코프에 해당 없으면 `-` |
| 관련 API | (예: /api/products, /api/tree/*) — 스코프에 해당 없으면 `-` |
| 관련 구현 경로 | (예: frontend/src/features/product/..., 매퍼: src/main/resources/mapper/ProductMapper.xml) |
| 참고 문서 | (예: docs/&lt;브랜치&gt;/product_base_info.md) |
```

## 산출물 구조

기능명세서는 `docs/template.md` 의 §1~§10 구조를 따라 구성됩니다. 해당 스코프에 해당하지 않는 섹션은 `"해당 없음 (스코프: X)"` 로 표기하고 생략합니다.

```
메타 헤더 (문서 최상단 표 — 스코프 포함)
§1  기능 개요
§2  데이터 모델        (트리/데이터 구조 · 테이블 스키마)
§3  화면 레이아웃      (ASCII 와이어프레임 · 패널/영역)
§4  기능 상세          (영역별 동작 · 툴바)
§5  API 스펙           (엔드포인트 · SQL · 검증 시나리오 · 에러 응답)
§6  비즈니스 규칙
§7  인증·보안
§8  기술 스택          — CLAUDE.md 참조로 생략 (명세서에 별도 기술 안 함)
§9  참고자료
§10 코드 품질 스킬      (feature-spec 자동 작성)
```

> §8 기술 스택은 프로젝트 공통(`CLAUDE.md`)을 그대로 따르므로 산출물에서 생략한다.
> 아래 "섹션별 작성 규칙" 은 산출물에서 분량이 큰 §2·§3·§4·§5 의 상세 작성 형식을 정의한다.

## 섹션 생략 기준 (기능 유형별 필수 매트릭스)

섹션을 "생략"할 때는 **자의적으로 판단하지 말고** 아래 매트릭스에 따른다.
매트릭스 표기: `필수` / `선택` / `-` (해당 없음) / `공통 참조` (CLAUDE.md로 대체)

컬럼은 `docs/template.md` 의 실제 섹션(§2~§7)이다. `§8 기술 스택`은 항상 CLAUDE.md 참조로 생략, `§9 참고자료`는 자료 유무에 따라 선택이므로 매트릭스에서 제외한다.

| 기능 유형 | §2 데이터 | §3 화면 | §4 기능 | §5 API | §6 규칙 | §7 인증 |
|---|---|---|---|---|---|---|
| **CRUD 화면** (목록/상세/편집) | 필수 | 필수 | 필수 | 필수 | 필수 | 공통 참조 |
| **순수 UI / 정적 화면** | - | 필수 | 필수 | - | 선택 | 공통 참조 |
| **계산/산출 로직** | 필수 | 선택 | 선택 | 필수 | 필수 | 공통 참조 |
| **백그라운드 작업/배치** | 필수 | - | 선택 | 선택 | 필수 | 공통 참조 |
| **외부 시스템 연동** | 선택 | 선택 | 선택 | 필수 | 필수 | 공통 참조 |
| **대시보드/통계** | 필수 | 필수 | 필수 | 필수 | 선택 | 공통 참조 |

> `공통 참조` = 프로젝트 공통 설정(`CLAUDE.md`)으로 대체, 기능 고유 요구사항이 있을 때만 작성.
> 매트릭스에 없는 유형이면 가장 유사한 행을 따르고, 이유를 명세서 상단 `참고 문서` 행에 한 줄로 명시한다.

## 섹션별 작성 규칙

### 데이터 모델 (§2)

이 기능에서 사용하는 데이터의 구조(트리/데이터 구조)를 정의합니다.

**포함 항목**:

- **테이블 스키마**: 컬럼명, 타입, 설명, 기능 내 역할을 표(table)로 작성
- **계층/관계 구조**: 테이블 간 관계나 데이터 계층을 텍스트 트리 다이어그램으로 표현
- **상세 테이블 매핑**: 데이터 유형별로 어떤 테이블과 JOIN하는지 매핑표 작성

| ... | ... | ... | ... |

### 계층 구조

(텍스트 트리 다이어그램)

### 상세 테이블 매핑

| 유형 | 상세 테이블 | JOIN 조건 |
|---|---|---|
| ... | ... | ... |
```

### API 스펙 (§5)

이 기능에서 사용하는 Backend API 엔드포인트를 정의합니다.

**각 API마다 포함할 항목**:

- **엔드포인트**: Method + Path + 인증 헤더
- **SQL 쿼리**: 해당 API가 실행하는 쿼리 (코드블록)
- **Response 예시**: 성공 시 JSON 응답 구조 (코드블록)
- **요청 본문** (POST/PUT인 경우): JSON 요청 구조
- **처리 순서** (복잡한 로직인 경우): 트랜잭션 내 처리 단계를 번호 목록으로 기술
- **에러 응답** (필수): 실패 조건별 HTTP 상태 코드 + 에러 코드 + 메시지 — 표로 작성
- **검증 시나리오** (필수): **정상 / 에지 / 거부** 3구분으로 각 케이스를 표로 작성. 자동화 테스트 시나리오 생성 시 참고용 필드로 사용된다.
- **규칙 요약**: 해당 API의 핵심 규칙을 불릿 포인트로 정리

**작성 형식**:

```markdown
### {번호}. {API 설명}

  {METHOD} {PATH}
  Authorization: Bearer {JWT}

(SQL 쿼리 코드블록)

**Response:**
(JSON 코드블록)

**처리 순서 (단일 트랜잭션):**
1. ...
2. ...

**에러 응답**

| 조건 | Status | 에러 코드 | 메시지 |
|---|---|---|---|
| (예: 필수 파라미터 누락) | 400 | INVALID_INPUT | ... |
| (예: 대상 없음) | 404 | NOT_FOUND | ... |
| (예: 권한 없음) | 403 | FORBIDDEN | ... |
| (예: 서버 오류) | 500 | INTERNAL_ERROR | ... |

**검증 시나리오**

| 구분 | 시나리오 | 입력 | 기대 결과 |
|---|---|---|---|
| 정상 | (예: 유효한 요청으로 조회) | ... | 200 + 데이터 N건 |
| 정상 | (예: 결과 0건인 정상 응답) | ... | 200 + 빈 배열 |
| 에지 | (예: 경계값 — 최대 길이 입력) | ... | 200 + 정상 처리 |
| 에지 | (예: 동시 요청으로 동일 데이터 수정) | ... | 낙관적 락 처리 |
| 거부 | (예: 필수 누락) | ... | 400 INVALID_INPUT |
| 거부 | (예: 토큰 만료) | ... | 401 UNAUTHORIZED |

**규칙 요약**
- ...
```

### 화면·기능 (§3·§4)

화면의 레이아웃(§3)과 영역별 기능·컴포넌트·동작(§4)을 정의합니다.

**포함 항목**:

- **화면 레이아웃**: ASCII 와이어프레임으로 전체 구조 표현
- **타이틀 바 스타일**: CSS 클래스 등 스타일 명세
- **패널 리사이즈/접기**: 동작 방식을 항목-값 표로 작성
- **영역별 상세**: 각 영역(패널, 툴바, 그리드 등)의 기능을 하위 섹션으로 분리
  - 검색/필터 툴바: 영역-구성-설명 표
  - 아이콘 툴바: 순서-아이콘-기능 표
  - 팝업/다이얼로그: 레이아웃(ASCII) + 데이터 소스 + 처리 로직 + 하단 버튼 + 규칙 요약 + 성공 후 동작 + 구현 파일 매핑
  - 그리드 컬럼: 컬럼키-헤더-너비-매핑 표
  - 동작 흐름: 번호 목록으로 사용자 액션 → 시스템 반응 기술
- **사용자 시나리오** (필수): 주요 기능(팝업/폼/그리드 편집 등)마다 **정상 / 에지 / 거부** 3구분의 시나리오 표. E2E 테스트 시나리오 생성 시 참고용 필드로 사용된다.
- **컴포넌트 목록**: 컴포넌트명-파일-역할 매핑표
- **체크박스/선택 동작**: 캐스케이딩 등 연동 규칙을 번호 목록으로 기술
- **키보드 네비게이션**: 키-조건-동작 표
- **스크롤/리사이즈**: 항목-설명 표
- **성능 최적화**: 적용된 최적화 기법을 항목-설명 표로 정리

**작성 형식**:

```markdown
### 화면 레이아웃 (ASCII 와이어프레임)

┌───────────────────────────────────┐
│  (전체 구조를 텍스트로 표현)        │

## 공통 작성 규칙

| 규칙 | 설명 |
|---|---|
| **마크다운 형식** | 모든 명세서는 `.md` 파일로 작성 |
| **표(table) 우선** | 정형 데이터는 표로, 흐름/순서는 번호 목록으로 |
| **코드블록 활용** | SQL, JSON, 실행 커맨드는 반드시 코드블록(```)으로 감싸기 |
| **ASCII 와이어프레임** | 화면 구조, 팝업 레이아웃은 텍스트 박스 문자(─ │ ┌ ┐ ├ ┤ └ ┘)로 표현 |
| **구현 파일 매핑** | 주요 기능마다 관련 파일 경로와 역할을 매핑표로 첨부 |
| **규칙 요약 패턴** | 복잡한 로직 뒤에는 반드시 "규칙 요약" 불릿 포인트를 추가하여 핵심을 정리 |
| **CLAUDE.md 중복 방지** | 프로젝트 공통 정보(기술 스택, 인증, 실행 커맨드 등)는 `CLAUDE.md` 참조로 대체 |
| **한국어 작성** | 모든 설명은 한국어, 기술 용어와 코드 식별자는 원문 유지 |

## ASCII 와이어프레임 작성 가이드

다른 Claude 인스턴스가 와도 동일한 스타일을 재현할 수 있도록, 와이어프레임은 다음 규칙을 따른다.

### 박스 문자

```
모서리:  ┌  ┐  └  ┘
수평선:  ─
수직선:  │
교차:    ├  ┤  ┬  ┴  ┼
```

### 치수 규칙

| 항목 | 규칙 |
|---|---|
| 최대 폭 | **80자** 이내 (가로 스크롤 방지) |
| 들여쓰기 | 들여쓰기 없음 (코드블록 내부에서 왼쪽 정렬) |
| 영역 라벨 | 각 영역 내부 **좌측 상단**에 `[영역명]` 형태로 표기 |
| 버튼 표기 | `[ 버튼명 ]` (양쪽 공백 포함) |
| 입력 필드 | `[_____________]` (밑줄로 길이 표현) |
| 체크박스 | `[ ]` / `[v]` **또는** `☐` / `☑` (둘 다 허용, 한 문서 내에서는 하나로 통일) |
| 드롭다운 | `[ 값 ▼ ]` |
| 생략 표시 | `...` (가로 생략) / `⋮` (세로 생략) |
| 노드 유형 아이콘 (선택) | `📁` 폴더 / `📦` 상품·그룹 / `📄` 문서·하위 항목 / `📌` 고정 항목 — 트리·목록 시각화에 사용 가능 |
| 상태 아이콘 (선택) | `✏` 수정 / `🗑` 삭제 / `➕` 추가 / `🔍` 검색 등 직관적 표현이 필요할 때 사용 |

> 이모지/아이콘은 **의미 전달을 돕는 보조 수단**으로만 사용한다. 박스 문자 골격은 항상 유지한다.

### 작성 예시

```
┌─────────────────────────────────────────────────────────────────┐
│ [타이틀 바]                                         [ 저장 ] [ X ] │
├─────────────────────────────────────────────────────────────────┤
│ [좌측 트리]          │ [우측 편집 패널]                            │
│ ┌─ 폴더A             │ ┌────────────────────────────────────┐    │
│ │  ├─ 상품1 [v]      │ │ 상품명: [___________________]      │    │
│ │  └─ 상품2 [ ]      │ │ 판매가: [___________________]      │    │
│ └─ 폴더B             │ │ 상태:   [ 판매중 ▼ ]               │    │
│                      │ └────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────────────┤
│                                            [ 취소 ]  [ 저장 ]    │
└─────────────────────────────────────────────────────────────────┘
```

## 용어 통일표

다른 Claude 인스턴스가 와서 작성해도 표기가 일관되도록, 용어는 **권장** 열을 따른다.
`대안` 열의 표기는 사용하지 않는다.

| 권장 | 대안 (사용 금지) |
|---|---|
| 팝업 | 다이얼로그, 모달, 창, 윈도우 |
| 그리드 | 테이블, 목록(데이터 표시용) |
| 패널 | 영역, 구역, 섹션(레이아웃 영역 지칭 시) |
| 컬럼 | 필드, 항목(테이블 컬럼 지칭 시) |
| 툴바 | 도구 모음, 툴박스 |
| 트리 | 트리뷰, 계층, 트리 구조 |
| 체크박스 | 체크, 선택상자 |
| 토스트 | 스낵바, 알림 배너 |
| 스피너 | 로더, 로딩 아이콘 |
| 드롭다운 | 셀렉트, 콤보박스 |
| 확인/취소 | OK/Cancel (한글 UI 우선) |
| 저장/삭제 | Save/Delete |
| 작업테이블 | 작업 테이블, 작업창 (작업/임시 테이블 지칭 시) |

> 신규 용어가 필요하면 명세서에 한 번만 정의하고 이후 일관되게 사용한다.
