00

이 문서가 하는 일

개발자가 누구든 명명·구조·계층 형태가 일정하게 나오도록 하는 것이 목적입니다. 여기 정리된 규약과 Claude 자산을 적용하면 그 범위가 유지됩니다.

스택은 공통, 애플리케이션은 환경이 다릅니다

이 문서의 적용 대상은 ILIS 편집시스템입니다. 규약의 업무 전제(배정 기반 단계 진행 등)도 그 기준입니다. 다만 ILIS 사업은 편집시스템·관리시스템·포털 세 시스템으로 구성되고 각자 배포됩니다. 이 자산을 다른 시스템에도 쓸 수 있게 하려면 환경에 따라 달라지는 것을 규약에서 분리해야 합니다. 그래서 여기 담긴 것은 환경이 달라져도 공통으로 쓸 수 있는 기술 선택과 가이드라인입니다.

예기준
규약에 있는 것 명명 규칙 · 패키지·폴더 구조 · 계층 책임 · 응답 형식 · 주석 태그 · 오류 처리 순서 환경이 바뀌어도 같다
규약에 없는 것 저장 매체 · 루트 경로 · 용량 제한 · 허용 확장자 · 서빙 방식 · DB 접속 정보 환경·배포마다 다르다
편집시스템 전제 배정 기반 단계 진행 · 200+FAIL 닫힌 목록 · JWT 인증 다른 시스템에 쓸 때 재검토

환경에 따라 달라지는 값은 빼는 것이 아니라 꽂을 자리를 정해둡니다. 규약은 ${FILE_ROOT} 처럼 주입 지점과 형식을 규정하고, 실제 값은 기능명세서와 환경별 설정 파일에서 정합니다.

그래서 이렇게 씁니다

규약에 답이 없다고 임의로 정하지 마세요. 환경 종속 값이면 기능명세서에서 정할 것이고, 형식·구조 문제인데 규약에 없다면 규약을 갱신할 대상입니다. 둘을 구분해서 알려주시면 됩니다.

02

개발 도구 설정

백엔드는 STS, 프론트엔드는 VS Code로 작업합니다. 두 IDE 모두 루트가 아니라 각자의 하위 폴더를 엽니다. STS로 루트를 import 하면 node_modules 인덱싱에서 멈추고, VS Code로 루트를 열면 프론트 확장이 Java 소스까지 훑습니다.

backend — STS

  • 워크스페이스: 저장소 바깥
  • 여는 법: backend/ import
  • 실행: localhost:8080

frontend — VS Code

  • 워크스페이스 개념 없음
  • 여는 법: frontend/ 폴더 열기
  • 실행: localhost:5173

STS 워크스페이스는 저장소 바깥에 둔다

VS Code는 폴더를 여는 것이 전부지만, STS의 워크스페이스는 소스 위치가 아니라 IDE 메타데이터 저장소입니다. 지정한 경로에 .metadata/를 만들고 인덱스·실행 설정을 쌓습니다. 절대경로와 로컬 환경이 박혀 있어 공유가 불가능하므로 저장소 안에 생기면 안 됩니다. 

# 워크스페이스 — git 과 무관, 개발자 개인 것
~/sts-workspace/
└── .metadata/

# 저장소 — git 으로 공유
~/git/ilis/
├── frontend/   ← VS Code 로 폴더 열기
└── backend/    ← STS 로 import

STS 실행 → 워크스페이스 선택: ~/sts-workspace
File → Import → Gradle(Maven) → Existing Project
  경로: ~/git/ilis/backend
  ☐ Copy projects into workspace   ← 반드시 해제
유의

Copy projects into workspace를 체크하면 워크스페이스로 복사본이 만들어집니다. 이 경우 편집한 내용이 git 저장소에 반영되지 않으므로 반드시 해제한 상태로 import 합니다.

로컬 개발 — Vite 가 API 를 프록시한다

운영에서는 nginx가 /api를 Spring으로 넘깁니다. 로컬에는 nginx가 없으므로 Vite dev 서버가 같은 역할을 합니다. 두 설정의 경로 규칙이 같으므로 프론트 코드는 어느 환경에서도 /api/...만 호출하면 됩니다. 

// frontend/vite.config.ts
server: {
  proxy: { '/api': { target: 'http://localhost:8080', changeOrigin: true } },
}

# deploy/nginx.conf.example (운영)
location / {
    try_files $uri $uri/ /index.html;   # SPA 새로고침 404 방지
}
location /api/ {
    proxy_pass http://127.0.0.1:8080/api/;
}
03

기술 스택

백엔드

기술무엇인가
Spring Boot 3.x 자바로 서버(API)를 만드는 프레임워크. 설정을 대신 해줘서 코드에 집중할 수 있다
JDK 17 이상 자바 개발·실행 환경. 17은 장기 지원(LTS) 버전이다
MyBatis SQL을 XML 파일에 직접 쓰고, 그 결과를 자바 객체에 담아주는 도구. SQL을 자동 생성하는 방식(JPA·Hibernate)과 달리 쿼리를 개발자가 직접 작성한다
REST HTTP 주소와 메서드(GET·POST·PUT·DELETE)로 기능을 표현하는 통신 방식. GET /api/v1/laws/1 이면 "1번 법령을 조회한다"는 뜻이다
PostgreSQL 17.x 관계형 데이터베이스(RDBMS). 데이터를 표(테이블) 형태로 저장하고 SQL로 조회한다
Lombok getter·생성자처럼 반복되는 자바 코드를 컴파일 시점에 자동으로 만들어주는 라이브러리
JWT 로그인한 사용자임을 증명하는 토큰. 서버가 발급하고, 이후 요청마다 함께 보낸다. (전 기능이 로그인 뒤라는 전제는 편집시스템·관리 기준이며, 대국민 포털의 공개 열람은 인증 없이 접근하므로 재검토 대상이다)

프론트엔드

기술무엇인가
React 화면을 컴포넌트(버튼·표·폼 같은 조각)로 나눠 조립하는 방식의 자바스크립트 라이브러리
TypeScript 자바스크립트에 타입을 붙인 언어. 오타나 타입 실수를 실행 전에 잡아준다
Vite 프론트엔드 빌드 도구. 두 가지 일을 한다 — 개발 중에는 코드를 저장하면 브라우저에 즉시 반영되는 개발 서버를 띄우고, 배포할 때는 흩어진 소스 파일을 브라우저가 읽을 수 있는 파일 몇 개로 묶어(build) 준다. 묶인 결과물(dist/)은 그냥 HTML·JS·CSS 파일이라 웹 서버에 올리기만 하면 된다
React Router 주소(URL)와 화면을 연결해주는 라이브러리. /laws/123 으로 들어오면 어느 화면을 보여줄지, 어떤 공통 레이아웃으로 감쌀지 정한다
shadcn/ui 버튼·입력창·표 같은 UI 부품 모음. 설치해서 가져다 쓰는 게 아니라 소스 코드를 우리 저장소로 복사해 온다. 복사된 뒤엔 우리 코드라 직접 고칠 수 있다
Tailwind CSS 미리 정의된 클래스 이름을 붙여 스타일을 입히는 CSS 도구 (class="flex gap-4 text-sm" 처럼 쓴다)
TanStack Query 서버에서 받아온 데이터의 조회·보관(캐시)·다시 불러오기를 대신 관리해주는 라이브러리. 로딩 중·오류·성공 상태를 알아서 알려준다
Zustand 화면 상태(로그인 사용자·사이드바 열림 여부·선택한 언어 등)를 담아두는 작은 저장소
axios 서버에 HTTP 요청을 보내는 라이브러리. 요청과 응답을 가로채(인터셉터) 공통 처리를 넣을 수 있다
React Hook Form 입력 폼의 값과 오류 메시지를 관리해주는 라이브러리
Zod 입력값 규칙(필수 여부·길이·형식)을 코드로 정의하고 검사하는 도구
react-i18next 화면 문구를 언어별 파일로 분리해 두고 전환해주는 다국어 라이브러리

배포·운영

기술무엇인가
nginx 웹 서버. 정적 파일(HTML·JS·CSS)을 브라우저에 내려주고, 특정 주소로 온 요청을 뒤에 있는 다른 서버로 전달(프록시) 한다
Git · GitLab 소스 코드 버전 관리 도구와 사내 저장소
용어

SPA (Single Page Application) — 페이지를 통째로 새로 불러오지 않고 화면 일부만 바꿔가며 동작하는 방식입니다. 처음 한 번 파일을 받아온 뒤로는 서버와 데이터만 주고받습니다.

공통 응답 포맷 — 이 프로젝트의 모든 API 응답이 갖는 고정된 겉껍질입니다. 성공이든 실패든 항상 status·code·message·data 네 칸을 갖고, 실제 데이터는 data 안에 들어 있습니다. 자세한 내용은 §04에 있습니다.

인터셉터(interceptor) — 요청이나 응답이 오갈 때 중간에서 가로채 공통 처리를 하는 장치입니다.

04

백엔드 규약

전문은 backend/CLAUDE.md에 있습니다. 여기서는 자주 걸리는 것만 추렸습니다.

패키지 구조 — global / domain 2분할

기술 계층을 최상위에 나열하지 않습니다. 도메인이 늘면 한 패키지에 파일이 몰립니다.

{base}
├── global       config · exception · response · util · security · aop
└── domain
    └── law      api · service · repository · domain · dto

{base} 는 프로젝트가 소유한 도메인의 역순으로 정합니다 (고정 표준이 아닙니다). 정부 시스템이라 예: id.go.peraturan.ilis (도메인 ilis.peraturan.go.id). 전문·규칙은 backend/CLAUDE.md.

판례 도메인은 precedent를 씁니다 — case는 Java 예약어라 패키지명으로 쓸 수 없습니다.

요청 한 건이 지나가는 경로

법령 등록을 예로 들면 파일이 이렇게 이어집니다.

POST /api/v1/laws                       ← 브라우저 요청 (nginx 프록시 경유)
   │
   ▼
LawController.createLaw()              domain/law/api/
   │  @Valid 로 요청 형식 검증. 업무 로직은 두지 않는다
   ▼
LawService.createLaw()                 domain/law/service/
   │  업무 규칙 검사 · @Transactional 로 트랜잭션 시작
   ▼
LawMapper.insertLaw()                  domain/law/repository/
   │  인터페이스만 있고 본체는 없다

Service — 인터페이스를 만들지 않는다

구현체가 하나뿐인 인터페이스는 파일만 늘립니다(Mockito는 클래스도 목킹합니다). 여러 구현체가 실제로 필요한 경우만 예외입니다.

  • 클래스에 @Transactional(readOnly = true), 쓰기 메소드에만 @Transactional
  • 트랜잭션 전파는 REQUIRED·REQUIRES_NEW 2개만 사용 (나머지 5종 금지)
  • 권한 체크를 Service 안에서 하지 않습니다 — Security 계층(@PreAuthorize)에서 차단

HTTP 상태 코드는 공통 응답 포맷과 별개로 판단합니다

기준은 요청 자체가 유효한가입니다 — 주소·형식·인증·대상 존재 여부 넷을 통과한 뒤 업무 규칙이 거절한 것만 200 + FAIL입니다.

상황HTTP예시
정상200 / 201조회 성공, 등록 완료
업무 규칙 거절200검수 미완료 건 개시 시도, 중복 등록, 기한 경과

업무 규칙 거절이 HTTP 200으로 나갑니다. 이는 업무 거절까지 4xx로 내리면 모니터링 지표에 오류가 쌓여 실제 장애가 묻히기 때문입니다. 대신 공통 응답 포맷의 code 기준 메트릭을 별도로 수집해야 합니다.

200 + FAIL 대상은 닫힌 목록 3종(상태 전이 불가·중복 등록·기한 경과)입니다. 여기 없는 사례가 생기면 규약을 먼저 갱신합니다.

이 3종 목록은 편집시스템 업무 기준입니다 (편집시스템 전제, §00 참조). 200+FAIL 로 내린다는 규칙 자체는 공통이지만 목록 내용은 시스템마다 다릅니다. 대국민 포털은 업무 거절이 거의 없고(검색·열람 중심), 관리 페이지는 거절 사례가 다르므로 각 시스템이 자기 닫힌 목록을 정의합니다.

목록 응답 — 배열을 그대로 내리지 않고 PageResponse 로 감싼다

목록은 data 에 배열을 바로 담지 않습니다. PageResponse<T> 로 감싸서 전체 건수·페이지 정보를 함께 내립니다. 합쳐지면 ApiResponse<PageResponse<T>> 가 됩니다. 이 구조가 없으면 프론트가 페이지네이션을 그릴 수 없습니다. 

// global/response/PageResponse.java
public record PageResponse<T>(
    List<T> content,        // 현재 페이지 항목
    int page,               // 0-based 페이지 번호
    int size,               // 페이지 크기
    long totalElements,     // 전체 건수
    int totalPages          // 전체 페이지 수
) {}

{
  "status": "SUCCESS", "code": "OK", "message": "...",
  "data": { "content": [ ... ], "page": 0, "size": 20, "totalElements": 137, "totalPages": 7 }
}
주의

페이징이 없는 짧은 목록(코드 목록 등)은 배열을 그대로 내려도 됩니다. 다만 같은 엔드포인트가 상황에 따라 배열과 PageResponse 를 번갈아 내리면 안 됩니다.

검증 실패는 errors[] 로 필드마다 내린다

요청 바디를 @Valid 로 검증하다 실패하면 MethodArgumentNotValidException 이 발생하고, 전역 핸들러가 이를 잡아 필드별 오류를 errors[] 로 변환합니다. Controller 에서 직접 조립하지 않습니다. 

// global/response/ErrorDetail.java
public record ErrorDetail(String field, Object rejectedValue, String message) {}

// GlobalExceptionHandler — @Valid 실패를 errors[] 로 매핑
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ApiResponse<Void>> handleValidation(
        MethodArgumentNotValidException ex, Locale locale) {
    List<ErrorDetail> errors = ex.getBindingResult().getFieldErrors().stream()
        .map(e -> new ErrorDetail(e.getField(), e.getRejectedValue(), e.getDefaultMessage()))
        .toList();
    return ResponseEntity.badRequest().body(ApiResponse.fail("INVALID_INPUT",
        messageSource.getMessage("error.INVALID_INPUT", null, locale), errors));
}
errors[] 필드내용
field검증에 걸린 입력 필드명 (lawNo)
rejectedValue거절된 실제 입력값
message사용자에게 보일 문구 — 언어별 번역문(아래 다국어 참조)

사용자 문구는 Accept-Language 로 언어를 골라 내린다

프론트가 모든 요청에 Accept-Language(id | en | ko)를 실어 보냅니다. 백엔드는 그 언어로 번역된 문구를 message·errors[].message 에 담고, 프론트는 받은 문구를 그대로 표시합니다. 자바 코드에 한글·인니어 문구를 하드코딩하지 않습니다 — 예외는 code 만 들고, 문구는 전역 핸들러가 MessageSource 에서 조회합니다.

프론트엔드 규약

전문은 frontend/CLAUDE.md에 있습니다.

작업 모델이 설계 전제입니다 (편집시스템 전제)

아래는 편집시스템의 업무 전제에 묶인 규약입니다 (§00 참조. 환경이 바뀌어도 같은 규약이 아닙니다). 편집시스템은 문서 하나의 한 단계에 한 명만 배정되고, 입력자 → 편집자 → 검수자로 넘어가는 직렬 진행이라 같은 화면을 두 사람이 동시에 만지지 않습니다. 그래서 편집시스템 프론트는 실시간 동기화·충돌 해결·자동 재조회를 구현하지 않습니다.

다른 시스템에선 재검토

대국민 포털(동시 열람 다수·검색 중심)과 관리 페이지(동시 편집 가능)는 동시성·재조회 요건이 다릅니다. 위 전제와 거기서 파생된 결정(실시간 동기화·충돌 해결·자동 재조회 미구현, refetchOnWindowFocus: false 등)은 그 시스템에 적용할 땐 재검토합니다. 이 문서 전체의 적용 대상이 편집시스템이라는 점은 §00에 있습니다.

공통 응답 포맷은 인터셉터 한 곳에서만 벗긴다

서버는 항상 { status, code, message, data } 형태로 응답합니다. 화면에서 실제로 필요한 건 data 안쪽뿐인데, 이걸 화면마다 벗기면 한 군데라도 빠뜨렸을 때 업무 거절(HTTP 200 + FAIL)이 성공으로 처리됩니다. 그래서 딱 한 곳에서만 벗깁니다.

어디에 있고 언제 도는가

src/lib/api-client.ts 에 있습니다. 이 파일은 앱이 시작될 때 한 번 읽히고, 그때 인터셉터가 등록됩니다. 이후 apiClient 로 나가는 모든 요청과 들어오는 모든 응답이 자동으로 이 함수를 거칩니다. 개발자가 매번 호출하는 코드가 아닙니다. 

// src/lib/api-client.ts  ← 프로젝트에 이 파일 하나만 있습니다

export const apiClient = axios.create({ baseURL: '/api' });

// ↓ 앱 시작 시 1회 등록. 이후 모든 응답이 여기를 지나갑니다
apiClient.interceptors.response.use((res) => {
  const body = res.data;                 // { status, code, message, data }
  if (body.status === 'FAIL') {
    throw new BusinessError(body.code, body.message, body.errors);
  }
  return body.data;                      // ← 여기서 벗겨서 data 만 넘김
});

  • Zustand — 로그인 사용자·사이드바 토글·선택 언어 등 서버가 모르는 것
  • 서버 데이터를 Zustand에 복사하지 않습니다. 두 곳이 어긋납니다
  • 변경 mutation 후 invalidateQueries를 반드시 호출합니다 — 자동 재조회를 껐기 때문입니다

날짜 — LocalDate에 new Date()를 쓰지 않는다

new Date('2026-07-20')은 UTC 자정으로 파싱되어 지역에 따라 하루 앞 날짜로 표시됩니다. 선고일자·공포일자처럼 시각이 없는 날짜에서 이 버그가 납니다. 

// ✗ 2026-07-19 로 표시될 수 있음
new Intl.DateTimeFormat(locale).format(new Date('2026-07-20'));

// ✓ 문자열을 분해해 로컬 날짜로 구성
const [y, m, d] = value.split('-').map(Number);
new Intl.DateTimeFormat(locale).format(new Date(y, m - 1, d));

금지

  • 서버 데이터를 Zustand에 복사
  • 컴포넌트에서 공통 응답 포맷 직접 해체 (res.data.data)
  • 사용자에게 보이는 문구 하드코딩 — 전 화면 다국어 대상입니다
  • any 사용 — 불가피하면 unknown 후 좁히기
  • 실시간 동기화·폴링 구현 (편집시스템 전제, 포털·관리는 재검토)

    UNIQUE·후보키, 테이블에서 의미가 큰 것 우선
  3. FK · 상태 컬럼
    외래키와 STATUS 성격의 컬럼
  4. 선택 컬럼
    입력 시점이 같은 것끼리 인접, NULL 빈도가 낮은 순서로. 감사 컬럼은 맨 뒤

공통 감사 컬럼 — 모든 업무 테이블에 둔다

코드 테이블·매핑 테이블도 예외가 아닙니다. 컬럼 이름·타입을 테이블마다 다르게 짓지 않습니다. 아래 네 컬럼을 맨 뒤에 같은 형태로 둡니다. 

created_at  TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,   -- 생성 시각
created_by  VARCHAR(50)  NOT NULL,                            -- 생성 계정
updated_at  TIMESTAMP,                                          -- 수정 시각 (최초엔 NULL)
updated_by  VARCHAR(50)                                         -- 수정 계정
  • created_by·updated_by 는 애플리케이션이 채웁니다. 트리거로 채우지 않습니다(아래 금지).
  • 삭제는 물리 삭제가 기본입니다. 이력이 필요한 테이블만 is_deleted BOOLEAN NOT NULL DEFAULT FALSE 로 논리 삭제하며, 한 테이블에서 두 방식을 섞지 않습니다.

타입 선정

  • 코드 컬럼이라도 자릿수가 바뀔 가능성이 있으면 VARCHAR — CHAR는 공백 패딩 때문에 비교가 어긋납니다
  • PK는 BIGINT GENERATED BY DEFAULT AS IDENTITY (SERIAL은 레거시)
  • 금액·비율은 NUMERIC(p,s) — 부동소수점을 쓰지 않습니다
  • boolean은 BOOLEAN — CHAR(1)에 'Y'/'N'을 쓰지 않습니다
  • JSON은 JSONB (인덱싱·연산 지원)
  • 파일은 DB에 넣지 않고 경로만 저장합니다

마이그레이션 — 논리 변경 하나에 파일 하나

하나의 변경을 여러 버전으로 쪼개지 않습니다. 테이블·제약·인덱스를 나눠 놓으면 중간 버전이 "테이블은 있는데 PK가 없는" 상태로 남습니다. PostgreSQL은 DDL도 트랜잭션으로 처리하므로 한 파일에 담아도 원자적으로 적용됩니다. 

resources/db/migration/
├── V1__create_law_schema.sql      # 초기 스키마 — 테이블+제약+인덱스 한 벌
├── V2__create_review_schema.sql   # 검수 도메인 한 벌
└── V3__add_law_note_column.sql    # 컬럼 추가 하나

초기 스키마처럼 규모가 큰 경우만 도메인 단위로 나눕니다(법령·판례·검수·배정). 파일 안에서는 테이블 → 제약(ALTER TABLE ADD) → 인덱스 순서를 지킵니다.

FK 컬럼에는 인덱스를 만듭니다 — PostgreSQL은 FK 인덱스를 자동 생성하지 않습니다.

이 파일들은 언제 실행되는가

직접 psql 로 돌리지 않습니다. 백엔드 앱이 뜰 때 Flyway 가 자동으로 실행합니다. 이미 적용된 파일은 건너뛰고, 새로 추가된 파일만 순서대로 반영합니다. 

backend 기동
   │
   ▼
Flyway 가 flyway_schema_history 테이블을 읽는다   ← 무엇까지 적용했는지 기록
   │
   ▼
아직 적용 안 된 V*.sql 만 번호 순서대로 실행       resources/db/migration/
   │
   ▼
PostgreSQL 스키마 반영 완료 → 앱 기동 계속
중요

한 번 적용된 마이그레이션 파일은 수정하지 않습니다. Flyway 가 파일 내용의 체크섬을 기록해 두기 때문에, 내용을 고치면 다음 기동 때 검증에 실패해 앱이 아예 뜨지 않습니다.

컬럼을 추가하거나 바꿔야 하면 새 파일(V4__add_law_note.sql)을 만듭니다. 아직 아무 환경에도 적용되지 않은 파일만 고칠 수 있습니다.

금지

대상사유
트리거로직이 DB에 숨어 추적이 안 됩니다. 감사 컬럼도 애플리케이션이 채웁니다
저장 프로시저업무 로직은 Service 계층에 둡니다
DB Link시스템 간 연계는 API로 합니다
SELECT *컬럼 추가 시 예기치 않은 동작

Claude 자산 사용법

저장소의 .claude/ 폴더에 스킬 14종·서브에이전트 4종·슬래시 커맨드 7종이 함께 배포됩니다. clone 하면 별도 설치 없이 바로 동작합니다.

이 저장소에 kit 을 처음 적용할 때 (최초 1회)

배포받은 claude-kit 을 프로젝트에 처음 심는 것은 프롬프트 하나로 합니다. 압축을 푼 claude-kit/ 을 프로젝트 안에 두고, 프로젝트 루트에서 Claude Code 에 이렇게 요청합니다. 

# 프로젝트 루트에서 Claude Code 에게
claude-kit/SETUP.md 대로 이 프로젝트에 설치해줘

그러면 SETUP.md 절차대로 .claude/(스킬·에이전트·커맨드·컨텍스트·템플릿)·backend/CLAUDE.md·frontend/CLAUDE.md· 전역 규정이 배치되고, 수동으로 채워야 하는 것(builder 에이전트·settings.json·빌드도구/DBMS 미확정 값)이 함께 안내됩니다. 경로는 SETUP.md 가 실제 환경을 확인해 맞춥니다 (고정 경로를 가정하지 않습니다).

한 번만

이 적용은 저장소에 최초 1회입니다. 한 번 적용해 커밋해두면, 이후 팀원은 clone 만으로 .claude/ 자산이 따라와 바로 동작합니다.

커맨드 7종 — /mainline:이름 으로 실행

커맨드는 자동 발동하지 않습니다. /mainline: 뒤에 이름을 쳐서 명시적으로 실행하며, 정해진 절차(브랜치·커밋·MR 등)를 수행합니다.

커맨드하는 일
/mainline:start-feature한글 작업명을 받아 feature 브랜치 생성·전환 (영문 kebab slug 변환). 예: /mainline:start-feature 로그인 검증 추가
/mainline:implement-ticket이슈 번호·URL·요구사항 텍스트를 받아 구현 (feature 브랜치 전제). GitLab MCP 또는 glab 필요
/mainline:review-risk커밋 전 변경사항(staged·unstaged·untracked)을 리뷰
/mainline:commit변경을 분석해 커밋 메시지 생성 후 커밋
/mainline:commit-split변경을 기능·목적별로 나눠 여러 커밋으로 분리
/mainline:fix-failing-test실패 테스트의 원인(제품·테스트·환경)을 분류해 최소 수정으로 복구
/mainline:merge-request통합 브랜치로 rebase 후 GitLab MR 생성. origin·GitLab 필요

권장

코드를 작성하기 전에 규약을 외울 필요는 없습니다. 평소처럼 요청하면 규약이 적용된 결과가 나옵니다. 다만 결과가 규약과 맞는지는 리뷰에서 확인해 주세요. 규약 자체에 문제가 있으면 해당 CLAUDE.md나 스킬을 고쳐 팀 전체에 반영합니다.

개인 설정

.claude/settings.json은 팀 공유 설정이라 커밋 대상입니다. 개인적으로 허용하고 싶은 명령 등은 .claude/settings.local.json에 두면 커밋되지 않습니다.

온보딩 절차

  1. 사전 설치
    JDK 17 이상 · Node.js 20 LTS 이상 · Git · STS · VS Code
  2. 저장소 clone
    git clone <repo> && cd ilis
  3. 환경 설정 파일 생성
    .env.example → .env, application-local.yml.example → application-local.yml 복사 후 값 입력. DB 접속 정보는 저장소에 포함되지 않습니다
  4. 백엔드 실행
    워크스페이스를 저장소 바깥으로 지정 → backend/ import (Copy projects into workspace 해제) → 프로파일 local로 Boot Run → localhost:8080
  5. 프론트엔드 실행
    frontend/ 폴더 열기 → npm install → npm run dev → localhost:5173
  6. 연동 확인
    화면 진입 후 /api 호출이 8080으로 전달되는지 확인
  7. 규약 확인
    작업할 영역의 CLAUDE.md를 한 번 읽어둡니다 (backend/ 또는 frontend/)
