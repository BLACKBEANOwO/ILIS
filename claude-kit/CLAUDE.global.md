# CLAUDE.md — 전역 규정

이 파일은 Claude가 매 대화마다 자동으로 읽고 반드시 따라야 하는 전역 규정입니다.

---

## 언어

- 모든 응답은 **한국어**로 작성한다.
- 커밋 메시지도 한국어로 작성한다.

## 응답 원칙

- 항상 **사실 기반**으로 응답한다.
- 추측하거나 확인되지 않은 내용을 말할 때는 반드시 **[추측]** 이라고 표시한다.
- 참고한 내용(파일, 문서, 코드 등)이 있으면 **무엇을 참고했는지 출처를 명시**한다.

## 작업 진행 방식

- **작업 시작 전**: 작업일지를 확인하여 이전 작업 맥락을 파악하고 이어서 진행한다. (자세한 절차는 아래 §작업일지 운영 참조)
- **수정 전**: 관련 코드를 반드시 먼저 읽고 파악한 후 작업한다.
- **일관성 유지**: 새로운 기능을 추가할 때 기존 코드에서 동일/유사한 작업이 어떤 패턴(API 호출 방식, 유틸 함수, 설정 등)으로 구현되어 있는지 먼저 확인하고, 동일한 패턴을 따른다.
- **계획 먼저**: 수정 범위에 대한 계획을 먼저 보여주고, 사용자 확인 후 작업을 진행한다.
- **스킬 안내**: 작업에 스킬(도구)을 사용할 경우, 어떤 스킬을 이용하는지 사전에 안내한다.
- **작업 완료 후**: 작업일지를 필수로 기록한다. (자세한 절차는 아래 §작업일지 운영 참조)
- **md 파일 생성 경로**: 새로 생성하는 모든 `.md` 파일은 프로젝트 루트의 `docs/` 폴더 하위에 생성한다. 단, `CLAUDE.md`(루트 고정)와 작업일지(`작업일지/` 폴더)는 예외로 한다.

## 작업일지 운영 (LLM Wiki 패턴)

작업일지는 **3계층 구조**로 운영한다 — 원본 일지(raw) / 위키(synthesis) / 인덱스(catalog).
세부 가이드라인·체크리스트는 프로젝트별 `작업일지/GUIDELINES.md`를 단일 출처로 따른다.

### 폴더 구조

```
작업일지/
├── INDEX.md                  ← 카탈로그 (날짜·태그·상태·브랜치·위키 인덱스)
├── GUIDELINES.md             ← 운영 가이드라인 (단일 출처)
├── template-worklog.md       ← 일지 포맷
├── 작업일지_YYYY-MM-DD.md    ← 시간순 raw 로그 (append-only)
├── 작업일지_YYYY-MM-DD_v2.md ← /v2 버전 작업
└── wiki/
    ├── INDEX.md              ← 위키 카탈로그
    └── <namespace>-<topic>.md ← 주제별 종합본 (누적 갱신)
```

### 세션 시작 시 (반드시)

1. **`작업일지/INDEX.md` 의 § 목록 + § 위키 먼저 읽기** — 직전 일지/맥락/진행 중 작업 파악.
2. 신규 작업 시 — 관련 위키 페이지가 있으면 우선 읽고, 그다음 최신 일지 1~2개만 읽는다. **과거 일지를 모두 읽을 필요 없다(위키가 종합본)**.
3. `_v2` 일지가 있는데 사용자가 `v2`를 명시하지 않았으면 **어느 버전인지 반드시 확인**한 뒤 진행. 추측 금지.

### 세션 종료 시 (반드시)

1. **일지 작성** — `작업일지_YYYY-MM-DD.md` (또는 `_v2.md`)에 추가. 포맷은 `template-worklog.md` 따른다.
2. **INDEX.md 갱신** — § 목록 + § 태그 인덱스 + (필요 시) § 상태 인덱스에 한 줄씩 추가.
3. **위키 갱신** — frontmatter `tags:` 중 위키 페이지가 있는 태그가 있으면 해당 위키의 § 핵심 결정 / § 관련 일지 / § 상태 / frontmatter `updated:` 갱신. 자세한 절차는 `작업일지/GUIDELINES.md` §3 참조.
4. **위키 신설** — 동일 태그로 일지 2개 이상 누적되었는데 위키가 없으면 신설 검토.

### 작성 원칙

- 본문은 **요약 / 내러티브 / 근거(Commands) / 미해결 TODO** 4축을 유지.
- 마크업 나열보다 **문장 흐름**으로 사고를 보여준다 (Karpathy 스타일).
- "왜 했는가"가 빠지지 않도록 한다 — 6개월 뒤 자기 자신이 읽어도 맥락이 복원되어야 한다.
- frontmatter `branch:` = `git branch --show-current`, `base_commit:` = 시작 시점 HEAD short SHA.

### 버전 관리

- `작업일지/` 폴더 전체(위키 포함)는 **git 추적 제외**. 개인 기록용이므로 커밋·push·MR에 포함하지 않는다.
- 필요 시 프로젝트 `.gitignore` 에 `작업일지/` 추가.

## 산출물 작성 규정

보고서·명세서·정의서·PPTX 등 **사용자 외의 독자(발주처·팀·상급자)가 읽는 문서**를 만들거나 고칠 때는
`~/.claude/document-standards.md` 를 **먼저 읽고** 그 규정을 단일 출처로 따른다.
(용도 확인 · 미확정 표기 · 부연은 괄호 · 보고는 서사 — 4개 절)

작업일지 등 내부 기록에는 적용하지 않는다.

## 기술 선택 시 권장사항 안내

- 프로젝트 구성 과정에서 실무에서 널리 사용되는 모범 사례(best practice)나 권장 구성이 있을 경우, 사용자에게 **선택지와 함께 추천 사항을 안내**한다.
- 사용자가 해당 기술을 모르더라도 이해할 수 있도록 **간단히 설명**하고, 확인 후 적용한다.

### 프로젝트 초기 구성 시 반드시 안내할 항목

1. **언어/런타임 버전** — LTS 여부, 호환성
2. **빌드 도구** — 각 도구의 장단점 비교
3. **프로젝트 구조** — 패키지/디렉토리 구조 패턴 (계층형, 도메인형 등)
4. **환경 분리** — 로컬(local), 개발(dev), 운영(prod) 환경별 설정 분리 방법
   - 설정 파일 분리 (예: application-{profile}.yml)
   - 환경변수 관리 방식
   - 프로파일 활성화 방법
5. **데이터베이스** — 로컬 개발용 DB vs 운영 DB 구성, 마이그레이션 도구
6. **API 문서화** — Swagger/OpenAPI 등 문서 자동화 도구
7. **로깅 전략** — 환경별 로그 레벨, 포맷, 출력 대상
8. **예외 처리 구조** — 공통 에러 응답 형식, 글로벌 예외 핸들러
9. **테스트 구성** — 단위/통합 테스트 프레임워크, 테스트용 DB 설정
10. **보안 기본 설정** — CORS, 인증/인가 방식 (JWT, Session 등)

## 금지 사항

- 요청받은 범위만 수정한다. 주변 코드를 임의로 리팩토링하거나 개선하지 않는다.
- 보안 민감 파일(`.env`, 인증 정보 등)은 절대 커밋하지 않는다.

## 에러 대응

- 에러 발생 시 무작정 재시도하지 않고, **원인을 먼저 분석**하여 보고한다.
