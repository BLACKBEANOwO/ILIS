변경사항을 기능/목적별로 분류하여 여러 개의 커밋으로 나누어 커밋한다.

작업 절차:
1. git status로 변경된 파일 목록을 확인한다.
2. git diff로 staged 및 unstaged 변경 내용을 확인한다.
3. 변경사항을 기능/목적별로 그룹으로 분류한다.
4. 각 그룹에 대해 커밋 메시지를 작성한다.
5. 분류 결과를 사용자에게 보여주고 확인을 받는다.
6. 확인 후 그룹별로 순서대로 staging → commit을 반복한다.

분류 기준 (우선순위 순):
1. 기능 단위 — 하나의 기능 추가/변경에 관련된 파일끼리 묶는다.
2. 변경 유형 — feat, fix, refactor, docs, test, chore 등 성격이 같은 변경끼리 묶는다.
3. 모듈/디렉토리 — 같은 모듈이나 컴포넌트에 속한 변경끼리 묶는다.

커밋 메시지 규칙:
Conventional Commits 형식을 사용한다 (feat, fix, refactor, docs, test, chore).
- 제목은 50자 이내, 본문은 필요한 경우에만 추가한다.
- 한글로 작성한다.
- "무엇을 왜 변경했는지"에 초점을 맞춘다.

주의사항:
- 변경사항이 없으면 빈 커밋을 만들지 않는다.
- .env, credentials.json 등 민감 파일은 staging하지 않고 경고한다.
- git add -A 대신 변경된 파일을 명시적으로 staging 한다.
- 하나의 파일이 여러 기능에 걸치는 경우, 가장 핵심적인 그룹에 포함시킨다.
