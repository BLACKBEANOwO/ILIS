현재 브랜치에서 아직 커밋되지 않은 변경사항(staged + unstaged + untracked)을 리뷰한다.

## 절차

1. `git status`로 변경된 파일 목록(staged / unstaged / untracked)을 확인한다.
2. `git diff HEAD`로 추적 중인 파일의 변경사항을 확인한다 (staged + unstaged 통합).
   - 필요 시 `git diff --staged`(staged만)와 `git diff`(unstaged만)로 구분해서 본다.
3. untracked 파일은 `Read` 툴로 직접 읽어 내용을 파악한다.
4. 변경된 파일을 읽고 아래 체크리스트에 따라 리스크를 평가한다.

## 리스크 체크리스트 (우선순위순)

1. **동작 회귀** — 기존 동작이 의도치 않게 바뀌는가? (API 계약, 반환값, 부수효과)
2. **데이터 손상** — DB 스키마 변경, 마이그레이션, 직렬화 포맷 변경이 안전한가?
3. **권한 및 보안** — 인증/인가 우회, 입력 검증 누락, 민감 정보 노출이 있는가?
4. **테스트 누락** — 변경된 로직에 대응하는 테스트가 추가/수정되었는가?
5. **운영 영향** — 성능 저하, 설정 변경, 하위 호환성 깨짐이 있는가?

## 심각도 기준

- **Critical**: 장애 또는 데이터 손실을 유발할 수 있음. 머지 차단.
- **High**: 특정 조건에서 오동작 가능. 수정 후 머지 권장.
- **Medium**: 개선이 필요하나 즉시 문제되지 않음. 후속 작업으로 처리 가능.
- **Low**: 스타일, 네이밍 등 사소한 개선 사항.

## 출력 형식

### 요약
변경 규모(파일 수, 라인 수)와 전체 리스크 수준(Critical/High/Medium/Low)을 한 줄로 요약.

### 이슈 목록
| 심각도 | 파일:라인 | 설명 | 체크리스트 항목 |
|--------|-----------|------|-----------------|
| ...    | ...       | ...  | ...             |

이슈가 없으면 "발견된 이슈 없음"으로 표기.

### 확인이 필요한 가정
리뷰어가 판단할 수 없어 작성자 확인이 필요한 사항을 나열.

### 권장 후속 조치
구체적인 수정 방향이나 추가 테스트 제안을 나열.
