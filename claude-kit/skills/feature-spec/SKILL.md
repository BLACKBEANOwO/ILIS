# 기능명세서 작성 스킬

## 목적

`docs/template.md` 템플릿을 따라 **대화형으로** 기능명세서를 작성하고, 최종 산출물을 `docs/<현재브랜치>/<기능명>.md` 로 생성한다.

## 저장 경로 규칙 (CLAUDE.md 준수)

- **템플릿/공용 문서**: `docs/template.md` (영구 위치, 브랜치 무관)
- **산출물**: `docs/<현재브랜치>/<기능명>.md`
  - `<현재브랜치>`는 `git branch --show-current` 결과를 그대로 사용한다 (슬래시 포함 시 중첩 폴더).
  - 폴더가 없으면 `mkdir -p`로 생성한다.

## 진행 규칙 (중요)

- **다른 정보의 md 파일을 읽을 때는 사용자의 확인을 받는다**. 단, 이 스킬이 직접 참조하는 다음 파일은 예외다.
  - `docs/template.md` (템플릿 — 질문 순서/내용)
  - `./output_format.md` (산출물 포맷 — 구조·필드 규칙, 이 스킬 폴더에 위치)
  - `./example.md` (공식 참조 샘플 — 작성 스타일·깊이 기준, 이 스킬 폴더에 위치)
- 섹션 순서대로 질문하되, 한 번에 **최대 2~3개**만 묻는다.
- 답변이 충분하면 다음 섹션으로 넘어간다.
- 답변이 부족하면 템플릿의 `💬 Claude 질문` 블록 질문으로 보완한다.
- **DB 접속정보**를 받으면 직접 접속하여 스키마를 조회하고 결과를 보여준다. DB 정보는 `CLAUDE.md` 를 우선 참조한다.
- 각 섹션이 채워질 때마다 **진행 상황을 요약**한다.
- 사용자가 답변 대신 "모르겠다/알아서 해줘" 라고 하면, 합리적인 기본값을 제시하고 **확인을 받은 뒤** 진행한다.

## 실행 절차

1. 현재 브랜치 확인: `git branch --show-current` 결과를 저장해 둔다 (저장 경로 계산용).
2. `docs/template.md` 를 읽어 템플릿 구조를 파악한다.
3. **§1 기능 개요 + 스코프** 질문 — 기능명 / 한줄 설명 / 필요 이유 / 주요 사용자 / **스코프**.
   - 스코프 선택지와 의미를 사용자에게 제시하고 선택받는다.
     - `full` — 프론트 + 백엔드 + DB 풀스택 (기본)
     - `frontend-only` — 정적 UI만 (백엔드 없음, 더미 데이터)
     - `frontend-with-mock` — 화면 + mock API (백엔드 미정)
     - `backend-only` — API 전용
   - 사용자가 스코프를 선택하지 않으면 기본값 `full` 로 진행한다.
4. **스코프에 따라 이후 섹션을 동적으로 결정**한다.

   | 스코프 | 진행할 섹션 |
   |---|---|
   | `full` | §2 → §3 → §4 → §5 → §6 → §7 → §9 → §10(자동) |
   | `frontend-only` | §3 → §4 → §7(선택) → §9 → §10(자동) |
   | `frontend-with-mock` | §3 → §4 → §5-1(간소) → §7(선택) → §9 → §10(자동) |
   | `backend-only` | §2 → §5 → §6 → §7 → §9 → §10(자동) |

   > §8 기술 스택은 `CLAUDE.md` 를 그대로 따르므로 모든 스코프에서 건너뛴다.
   > §10 적용 스킬은 **사용자에게 묻지 않고 스코프 기반으로 자동 작성**한다 (아래 "§10 자동 작성 규칙" 참조).
   > 건너뛰는 섹션은 **사용자에게 질문하지 않는다**. 명세서에도 "해당 없음 (스코프: X)" 으로 표기한다.

5. **§2 데이터 모델** 진행 시 (스코프 `full` / `backend-only`)
   - `CLAUDE.md` 의 DB 접속정보를 우선 참조한다.
   - 사용자가 별도 DB를 지정하지 않으면 기본 DB로 접속하여 테이블/컬럼을 조회한다.
   - 테이블 목록이 주어지지 않았다면, 접속하여 후보를 보여주고 사용자 확인을 받는다.

6. **§3 진행 시** (프론트 포함 스코프)
   - 3-1 레이아웃 패턴은 필수.
   - 3-5 UI 상태, 3-6 사용자 시나리오는 옵션 C 구현·feature-test에서 필수 참고되므로 반드시 채운다.

7. **§5 진행 시** (API 포함 스코프)
   - `full` / `backend-only`: 5-1 ~ 5-4 모두 진행 (검증 시나리오·에러 응답 필수).
   - `frontend-with-mock`: 5-1 API 목록만 진행 (mock 응답 형태 결정용).

   > **백엔드 섹션 작성 시 코드 품질 스킬 참조 (필수)**
   > §2 데이터 모델 / §5 API 스펙 / §6 비즈니스 규칙을 작성할 때 다음 3개 스킬의 패턴을 참고하여 명세를 구체화한다 (스코프 `full` / `backend-only` / `frontend-with-mock`).
   > - **`spring-boot-patterns`** — §2/§5/§6 작성 시 Controller/Service/Mapper(MyBatis) 계층·DTO/Records·예외 처리 형태와, 트랜잭션 경계·XML 매퍼의 조회 전략·N+1 회피·락 정책을 명세에 반영
   > - **`api-contract-review`** — §5 작성 시 HTTP 동사·상태 코드·버저닝·DTO vs 엔티티·에러 포맷 검증
   > - **`architecture-review`** — §4/§6 작성 시 패키지 경계·레이어 분리·의존성 방향을 반영
   >
   > 각 스킬의 인스트럭션은 SKILL.md(영문)에 있다. 명세에 패턴을 적을 때 "왜 그렇게 하는지" 짧은 근거를 같이 적으면 구현 단계에서 트레이드오프가 명확해진다.

8. **§10 적용 스킬 자동 작성** — 사용자에게 묻지 않고 스코프 기준으로 다음을 명세서에 채운다.
   - `docs/template.md` §10 의 "스코프별 적용 매트릭스" 표를 그대로 복사하여 명세서에 반영한다.
   - 10-1 (명세 참조 스킬) / 10-2 (구현 적용 스킬) / 10-3 (스코프 매트릭스) 세 부분 모두 포함.
   - 기능 특성상 **추가/제외**할 스킬이 있다면 표 아래에 "사유: ..."를 한 줄 명시한다 (없으면 생략).
   - 이 섹션은 `feature-implement` 가 어떤 스킬을 트리거할지 결정하는 근거로 쓰이므로 비워두지 않는다.

9. 모든 섹션이 채워지면 **기능명세서 초안**을 작성한다.
   - **구조/필수 필드 규칙**: `.claude/skills/feature-spec/output_format.md` 를 따른다.
   - **작성 스타일/깊이/표기**: `.claude/skills/feature-spec/example.md` (공식 샘플)를 기준점으로 삼는다.
   - 두 문서 간 충돌이 있으면 포맷 규칙(`output_format.md`)을 우선한다.
   - 생략 섹션은 "해당 없음 (스코프: X)" 으로 명시적으로 표기한다.
10. 초안을 사용자에게 보여주고 피드백을 받는다.
11. 피드백 반영 후 최종본을 `docs/<현재브랜치>/<기능명>.md` 로 저장한다.
    - 저장 전 `mkdir -p docs/<현재브랜치>/`를 실행해 폴더가 있는지 보장한다.
    - `<기능명>` 은 §1에서 받은 기능명을 파일명 친화적으로 변환한다 (공백 → `_`).
    - 동일 파일이 이미 존재하면 **덮어쓰지 않고** 사용자에게 처리 방법을 확인한다 (덮어쓰기 / 이름 변경 / 취소).
12. **명세서 리뷰 게이트 (Codex adversarial-review)** — 아래 "명세서 리뷰 게이트" 절차를 수행한다. 리뷰 결과에 따라 명세서를 수정한 경우 저장(11번)과 리뷰(12번)를 반복한다.
13. 완료 후 다음 단계를 안내한다.

## 명세서 리뷰 게이트

저장된 기능명세서에 대해 Codex adversarial review를 **자동**으로 실행한다.

1. 다른 관계없는 working-tree 변경이 많이 섞여있는지 확인한다 (`git status --short`). 섞여있다면 사용자에게 고지한다: "리뷰 대상이 명세서 외 파일을 포함할 수 있습니다. 그대로 진행/중단/stash 중 선택해 주세요."
2. 다음 명령을 Bash 도구로 실행한다 (포그라운드 대기).
   ```bash
   node "$HOME/.claude/plugins/marketplaces/openai-codex/plugins/codex/scripts/codex-companion.mjs" adversarial-review --wait "기능명세서(docs/<브랜치>/<기능명>.md)의 모호한 요구사항·누락된 에지 케이스·구현 불가 조항·위험한 가정·스코프/데이터/API/인증 일관성을 찾아줘. 구현 전 단계의 문서 리뷰이며, 설계 결함 위주로 보고해줘."
   ```
   > companion 플러그인이 없거나 기본 codex 모델이 ChatGPT 계정 미지원이면 `codex exec --skip-git-repo-check -m gpt-5.4 "<프롬프트>"` 로 대체 실행한다.
   - `<브랜치>` / `<기능명>` 은 실제 값으로 치환하여 명령을 구성한다.
3. Codex stdout(영어 원문, JSON 구조 또는 서술)을 받으면 **한국어로 번역**하여 사용자에게 보고한다.
   - 본문에 **한국어 번역본**을 기본 표시하고, 원문은 `<details><summary>원문 (English)</summary> ... </details>` 로 접어서 함께 제공한다.
   - 발견사항·강도·파일 경로·라인 번호·심각도 라벨은 원문 그대로 보존한다. 요약·생략·재분류 금지.
   - 번역 과정에서 의미가 모호하면 원문 문구를 괄호로 병기한다.
4. 이슈 분류에 따라 흐름을 결정한다.
   - **이슈 없음 / 경미(`approve` 혹은 blocking-risk 없음)**: "명세서 리뷰 이상 없음 (혹은 경미)"라고 한 줄 요약 후 단계 13(완료 안내)로 진행.
   - **심각 이슈 발견**: 완료 안내로 넘어가지 **않는다**. 사용자에게 다음을 명시해 질문한다.
     - 발견된 심각 이슈 목록 (Codex 번역본 인용)
     - 명세서 반영 제안 (수정 문구/추가 섹션/제거 대상)
     - 질문: "명세서를 수정할까요? (수정 반영 / 이대로 완료 / 리뷰 무시 사유 기록)"
   - **사용자 승인을 받은 뒤에만** 명세서를 수정하고 단계 11(저장)부터 다시 수행한다. 수정 반영 후 본 리뷰 게이트를 다시 통과해야 한다.
5. 리뷰가 최종 통과하면 명세서 메타 헤더 `참고 문서` 행 또는 본문 말미에 `Codex adversarial review 통과 (YYYY-MM-DD)` 한 줄을 남긴다 (이후 feature-implement에서 이력 확인용).

## 완료 시 안내 메시지

기능명세서 생성이 끝나면 사용자에게 다음을 안내한다.
**후속 단계(구현, 테스트 등)는 자동으로 진행하지 않는다.** 사용자의 명시적 지시가 있을 때만 진행한다.

```
기능명세서가 완성되었습니다 → docs/<현재브랜치>/<기능명>.md

이후 단계는 사용자의 명시적 지시가 있을 때만 진행됩니다.
- 구현: 별도로 "이 명세서를 구현해줘" 등으로 요청해 주세요.
- 브라우저 테스트: 구현 완료 확인 후, 직접 `feature-test` 스킬을 트리거해 주세요.
```

## 주의사항

- **요청 범위를 벗어난 작업은 하지 않는다**. 기능명세서만 작성하며, 명시적인 요청 없이 코드 수정/구현을 하지 않는다.
- 사용자가 제공하지 않은 정보를 추측으로 채워 넣지 않는다. 불명확하면 질문한다.
- 추측으로 내용을 채우는 경우에는 반드시 `[추측]` 표시를 붙인다.
- 참고한 파일(스키마, 기존 명세서 등)의 출처를 명세서에 명시한다.
- **외부 라이브러리·프레임워크·SDK·API·CLI 도구(React, React Hook Form, Zod, Spring Boot, Spring Security, MyBatis, PostgreSQL 등)의 문서·API·설정값·버전별 동작·마이그레이션 가이드가 필요하면 반드시 `use context7` 로 최신 공식 문서를 조회한 뒤 명세서에 반영한다.** 훈련 데이터의 구버전 정보로 작성하지 않는다. (CLAUDE.md "문서 참조 규칙" 참조)
