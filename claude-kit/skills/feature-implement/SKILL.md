# 기능 구현 스킬

## 목적

기능명세서(`docs/<브랜치>/<기능명>.md`)를 기반으로 **단계적으로** 기능을 구현한다. 각 단계는 서브에이전트(`frontend-builder`, `backend-builder`)에 위임한다. **화면 확인 게이트가 있는 흐름(스코프 `full`의 옵션 A/C, `frontend-only`, `frontend-with-mock`)에서는 사용자 승인 전까지 이후 단계로 넘어가지 않는다.** (스코프 `backend-only`와 옵션 B는 화면 확인 게이트가 없거나 순서가 다르므로 각 흐름의 게이트 위치를 따른다.)

> **전제 (중요)**: 이 스킬은 `frontend-builder`·`backend-builder` 서브에이전트를 전제로 한다. ILIS용으로 두 에이전트를 먼저 정의(React 프론트 · Spring Boot 3 + MyBatis 백엔드 스택)해야 동작한다. 두 에이전트가 없으면 위임 단계가 실패한다.

## 스코프 × 옵션

기능명세서 §1의 **스코프** 에 따라 사용할 구현 흐름이 결정된다. 옵션은 스코프가 풀스택(`full`)일 때만 실질적인 선택지가 된다.

| 명세서 스코프 | 사용 흐름 | 옵션 선택 |
|---|---|---|
| **`full`** | 프론트 + 백엔드 양쪽 구현 | 옵션 A / B / **C(기본)** 중 선택 |
| **`frontend-only`** | 프론트 UI만 (백엔드 없음) | 흐름 고정 — UI 구현(3-1) + Codex 리뷰(3-1R) + 화면 확인 게이트(3-2) 후 종료 |
| **`frontend-with-mock`** | 프론트 + mock API | 옵션 A 고정 (단, 백엔드 구현 단계는 **수행하지 않음**) |
| **`backend-only`** | 백엔드 + DB만 | 흐름 고정 — `[backend-builder · full]` 단독 실행 |

### 옵션 (스코프 `full` 전용)

| 코드 | 이름 | 흐름 |
|---|---|---|
| **C (기본)** | UI 더미 → 백엔드 → 실 API 전환 | 프론트를 정적 더미 데이터로 먼저 만들어 화면만 확인 → 백엔드 전체 구현 → 프론트를 실 API 연동으로 전환 |
| A | Mock API | 프론트는 MSW 등 mock으로 구현 → 화면 확인 → 백엔드 구현 + mock 제거 |
| B | 백엔드 스켈레톤 먼저 | 백엔드 스켈레톤(빈 응답) → 프론트 실 API 연동 → 화면 확인 → 백엔드 로직 채우기 |

### 옵션 선택 가이드 (스코프 `full`)

- **iteration 2~5회 예상** → C (기본값)
- **한 번에 승인 가능한 단순 기능** → B
- **중간 수정 2~3회 예상, 백엔드 계약 안정적** → A

## 진행 규칙 (중요)

- 사용자가 지정한 기능명세서 파일은 그대로 읽는다. 그 외 다른 md 파일 열람은 사용자 확인을 받는다.
- 각 구현 단계는 **서브에이전트에 위임**한다. 메인은 위임·중개·검증만 담당한다.
- **화면 확인 게이트**는 명시적 승인이 있을 때만 통과한다. 승인 전까지 이후 단계를 진행하지 않는다.
- 서브에이전트 완료 보고는 각 에이전트 정의의 **완료 보고 포맷**을 준수해야 한다. 누락 시 재요청.
- 단계마다 작업 범위를 서브에이전트에게 **명시적으로 지시**한다 (예: `ui-only-dummy`, `skeleton`, `full`).
- **Codex 리뷰 게이트**: 각 서브에이전트 단계가 끝날 때마다 Codex 리뷰를 자동 실행한다 (아래 "Codex 리뷰 게이트" 절 참조).

## Codex 리뷰 게이트

각 서브에이전트(`[frontend-builder ...]` / `[backend-builder ...]`) 위임이 끝날 때마다 아래 절차를 **자동**으로 수행한다.

1. 서브에이전트 완료 보고를 수신한 직후, 다음 명령을 Bash 도구로 실행한다 (포그라운드 대기).
   ```bash
   node "$HOME/.claude/plugins/marketplaces/openai-codex/plugins/codex/scripts/codex-companion.mjs" review --wait
   ```
   > companion 미설치/기본 codex 모델 미지원 시 `codex exec --skip-git-repo-check -m gpt-5.4 "<변경 리뷰 프롬프트>"` 로 대체한다. (이 프로젝트에서 gpt-5.3/5.4-codex 미지원, gpt-5.4 동작 확인)
2. Codex stdout(영어 원문)을 받으면 **한국어로 번역**하여 사용자에게 보고한다.
   - 본문에 **한국어 번역본**을 기본 표시하고, 원문은 접은 블록(`<details><summary>원문 (English)</summary> ... </details>`)으로 함께 제공한다.
   - 발견사항·경고·권고의 **내용·강도·파일 경로·라인 번호·심각도 라벨**은 원문 그대로 보존한다. 요약·생략·재분류 금지.
   - 번역 과정에서 의미가 모호하면 원문 문구를 괄호로 병기한다 (예: "경합 상태(race condition)").
3. 이슈 분류에 따라 흐름을 결정한다.
   - **이슈 없음 / 경미**: "Codex 리뷰 이상 없음 (혹은 경미)"라고 한 줄로 보고한 뒤 다음 단계로 진행.
   - **심각 이슈 발견**: 다음 단계(다음 서브에이전트·통합 확인·화면 확인 게이트 등)로 **넘어가지 않는다**. 사용자에게 다음을 명시해 질문한다.
     - 발견된 심각 이슈 목록 (Codex 원문 인용)
     - 수정 방향 제안
     - 질문: "이대로 수정 위임해도 될까요? (승인 / 직접 수정 / 무시하고 진행)"
   - **사용자 승인을 받은 뒤에만** 해당 단계의 서브에이전트에게 수정 범위를 명시해 재위임한다. 재위임 후에도 동일한 Codex 리뷰 게이트를 다시 통과해야 한다.
4. 통합 확인 단계(`단계 3-x 통합 확인`)처럼 **서브에이전트 위임이 아닌 단계**에는 Codex 리뷰를 적용하지 않는다.
5. 화면 확인 게이트 이전의 Codex 리뷰가 통과해야 화면 확인 요청이 의미 있으므로, **화면 확인 게이트는 Codex 리뷰 이후**에 올린다.

## 실행 절차

### 단계 1 — 사용자 질의

다음을 확인한다 (이미 알려진 정보는 건너뛴다).

| 항목 | 필수 | 기본값 |
|---|---|---|
| 기능명세서 경로 | 필수 | - |
| 구현 전략 옵션 (A/B/C) | 스코프가 `full`일 때만 | C |
| 수정 반복 예상 횟수 (옵션 선택 가이드에 활용) | 선택 | 2~5회 |

### 단계 2 — 기능명세서 확인 및 스코프 판별

1. 사용자가 지정한 명세서를 읽는다.
2. 메타 헤더의 **스코프** 필드를 확인한다 (없으면 사용자에게 질문).
3. 스코프에 따라 필요한 섹션의 존재/누락을 확인한다.
   - `full` · `backend-only`: §2, §5 필수
   - `full` · `frontend-only` · `frontend-with-mock`: §3, §4 필수
   - `frontend-with-mock`: §5-1 API 목록 필요
4. 구현 범위를 사용자에게 요약하여 보여준다 (스코프 + 어느 단계까지 진행되는지).
5. 사용자 확인을 받은 뒤 다음 단계로 진행한다.

### 단계 3 — 스코프별 구현 흐름

> 스코프에 따라 아래 중 **하나의 흐름만** 수행한다. 스코프와 무관한 단계는 실행하지 않는다.

#### 스코프 `frontend-only` — UI만 구현 후 종료

```
단계 3-1 [frontend-builder · ui-only-dummy]
  - 기능명세서 §3 · §4 전달
  - 정적 더미 데이터로 UI + 인터랙션 구현 (API 호출 전혀 없음)
  - §3-5(UI 상태), §3-6(사용자 시나리오) 모두 충족
  - 완료 보고 포맷 준수

🔍 단계 3-1R Codex 리뷰 (--wait)
  - "Codex 리뷰 게이트" 절차 적용
  - 심각 이슈 → 사용자 승인 후 단계 3-1 재위임, 통과할 때까지 반복

🛑 단계 3-2 화면 확인 게이트 (최종)
  - 확인 요청 후 사용자 승인으로 스킬 종료
  - 수정 요청 시 단계 3-1로 되돌아감
```

#### 스코프 `frontend-with-mock` — 화면 + mock API로 종료

```
단계 3-1 [frontend-builder · mock-api]
  - 기능명세서 §3 · §4 · §5-1 전달
  - MSW 등 mock 레이어 설정 + 실 API 호출 코드 작성 (mock이 응답)
  - 완료 보고 포맷 준수

🔍 단계 3-1R Codex 리뷰 (--wait)
  - "Codex 리뷰 게이트" 절차 적용
  - 심각 이슈 → 사용자 승인 후 단계 3-1 재위임, 통과할 때까지 반복

🛑 단계 3-2 화면 확인 게이트 (최종)
  - 사용자 승인 시 종료
  - mock → 실 API 전환은 이 스킬의 범위 밖 (이후 별도 요청 시 백엔드 구현 + 전환)
```

#### 스코프 `backend-only` — 백엔드만 구현

```
단계 3-1 [backend-builder · full]
  - 기능명세서 §2 · §5 · §6 · §7 전달
  - Controller + Service + Mapper(MyBatis) 로직 + 스키마 마이그레이션(도구는 프로젝트 규약)까지 전체 구현
  - 검증 시나리오(§5-3)·에러 응답(§5-4)을 포함한 완결 구현
  - 완료 보고 포맷 준수

🔍 단계 3-1R Codex 리뷰 (--wait)
  - "Codex 리뷰 게이트" 절차 적용
  - 심각 이슈 → 사용자 승인 후 단계 3-1 재위임, 통과할 때까지 반복

단계 3-2 통합 확인
  - actuator 헬스체크 + curl 예시로 핵심 엔드포인트 검증
  - 결과 요약 보고
```

#### 스코프 `full` — 옵션 A/B/C 중 선택 (기본 C)

##### 옵션 C (기본): UI 더미 → 백엔드 → 실 API 전환

```
단계 3-1 [frontend-builder · ui-only-dummy]
  - 기능명세서 §3 Frontend를 전달
  - 정적 더미 데이터로 UI만 구현 (API 호출 없음)
  - 주요 인터랙션(검색/정렬/팝업 등)은 더미 데이터 내에서 동작 가능하도록 구현
  - 완료 보고 포맷 준수

🔍 단계 3-1R Codex 리뷰 (--wait)
  - "Codex 리뷰 게이트" 절차 적용
  - 심각 이슈 → 사용자 승인 후 단계 3-1 재위임, 통과할 때까지 반복

🛑 단계 3-2 화면 확인 게이트
  - 사용자에게 확인 요청 항목 제시:
    · 실행 URL (프론트 개발 서버 URL — CLAUDE.md/프로젝트 규약 참조)
    · 주요 화면 스크린샷 설명
    · 구현된 인터랙션 목록
    · 알려진 제약(더미 데이터, TODO 등)
  - 질문: "화면 형태와 인터랙션을 확인해 주세요. 승인하시면 백엔드 구현으로 넘어가겠습니다."
  - 승인 전까지 대기. 수정 요청이 오면 단계 3-1로 돌아간다.

단계 3-3 [backend-builder · full]
  - 기능명세서 §2 Backend API 전달
  - Controller + Service + Mapper(MyBatis) 로직 + 스키마 마이그레이션(도구는 프로젝트 규약)까지 전체 구현
  - 완료 보고 포맷 준수

🔍 단계 3-3R Codex 리뷰 (--wait)
  - "Codex 리뷰 게이트" 절차 적용
  - 심각 이슈 → 사용자 승인 후 단계 3-3 재위임, 통과할 때까지 반복

단계 3-4 [frontend-builder · real-api]
  - 더미 데이터 제거
  - 실 API 연동 (`/api/*` 상대경로 등 프로젝트 API 연동 방식)
  - 완료 보고 포맷 준수

🔍 단계 3-4R Codex 리뷰 (--wait)
  - "Codex 리뷰 게이트" 절차 적용
  - 심각 이슈 → 사용자 승인 후 단계 3-4 재위임, 통과할 때까지 반복

단계 3-5 통합 확인
  - 메인이 최종 동작(프론트 ↔ 백엔드)을 확인
  - 결과 요약을 사용자에게 보고
```

🔍 단계 3-4R Codex 리뷰 (--wait)          → "Codex 리뷰 게이트" 절차
단계 3-5 통합 확인
```

##### 옵션 B: 백엔드 스켈레톤 먼저

```

단계 3-2 [frontend-builder · real-api]    → 실 API 호출 (빈 응답 허용)

🔍 단계 3-4R Codex 리뷰 (--wait)          → "Codex 리뷰 게이트" 절차
단계 3-5 통합 확인
```

### 단계 4 — 완료 보고

사용자에게 다음을 안내한다.
**후속 단계(테스트, 커밋 등)는 자동으로 진행하지 않는다.** 사용자의 명시적 지시가 있을 때만 진행한다.

```
<기능명> 구현이 완료되었습니다.

- 전략: 옵션 <X>
- 변경 파일: N개
- 새로 추가된 엔드포인트: N개
- 스키마 마이그레이션: (있으면 파일명)

## 서브에이전트 호출 규약

각 서브에이전트에 위임할 때 다음 정보를 명시적으로 전달한다.

```
[대상] frontend-builder 또는 backend-builder
[작업 범위] ui-only-dummy / mock-api / real-api / skeleton / full / logic-only
[기능명세서 경로] docs/<브랜치>/<기능명>.md
[참조 섹션] §3 Frontend / §2 Backend API
[제약] (추가 제약이 있으면 여기에)
[완료 보고] 에이전트 정의의 완료 보고 포맷 준수
```

## 산출물

- 프론트 변경 파일 (frontend/...)
- 백엔드 변경 파일 (backend/...)
- 스키마 마이그레이션 파일 (백엔드 구현 단계, 도구는 프로젝트 규약)
- MyBatis Mapper XML (SQL) 변경 파일 (백엔드 구현 단계)

## 주의사항

- **요청 범위를 벗어난 수정은 하지 않는다** — 명세서에 없는 기능 추가나 주변 리팩토링 금지.
- **추측으로 구현하지 않는다** — 명세서가 불명확하면 사용자에게 질문한다. 추측으로 채울 경우 `[추측]` 표기.
- **보안 민감 정보**(비밀번호, 토큰, 내부 엔드포인트)를 코드/주석에 남기지 않는다.
- **화면 확인 게이트 우회 금지** — 사용자 승인 없이 백엔드 단계로 넘어가지 않는다.
