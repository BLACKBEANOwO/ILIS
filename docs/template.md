# 기능명세서 작성 템플릿

> (설치 시 이 파일은 프로젝트 루트 `docs/template.md`로 위치한다. kit에서는 스테이징 경로에 둔다.)

> **이 문서의 목적**: 하네스 엔지니어링용 **기능명세서**를 일관된 품질로 생성하기 위한 템플릿입니다.
>
> **작성 대상**: 개발자 · 기획자 모두 사용 가능

### 파일 위치 규칙

- **이 템플릿 파일**: `docs/template.md` (브랜치 무관, 영구 위치)
- **작성된 기능명세서**: `docs/<현재브랜치>/<기능명>.md`
  - `<현재브랜치>`는 `git branch --show-current` 결과를 그대로 사용한다 (슬래시 포함 시 중첩 폴더).
  - 폴더가 없으면 `mkdir -p`로 생성한다.
  - 예: 브랜치가 `feature/agent-studio`이면 `docs/feature/agent-studio/<기능명>.md`

---

### 스코프(scope) 선택

기능의 범위에 따라 작성해야 하는 섹션이 달라집니다. §1에서 스코프를 먼저 고르면, 관련 없는 섹션은 비워두어도 됩니다.

| 스코프 | 설명 | 필수 섹션 |
|---|---|---|
| **`full`** (기본) | 프론트 + 백엔드 + DB 풀스택 | §1 · §2 · §3 · §4 · §5 · §6 · §7 |
| **`frontend-only`** | 정적 UI만 (백엔드 없음, 더미 데이터) | §1 · §3 · §4 |
| **`frontend-with-mock`** | 화면 + mock API (백엔드 미정·나중에 구현) | §1 · §3 · §4 · §5 (간소) |
| **`backend-only`** | API 전용 (다른 클라이언트가 사용) | §1 · §2 · §5 · §6 · §7 |

각 섹션 헤더에는 `[스코프: ...]` 표기로 해당 스코프에서만 채우면 되는지 알려줍니다.

> **§5-7 파일 업로드·다운로드**는 스코프와 무관하게 **파일을 다루는 기능이면 채웁니다.**
> (`frontend-only` 라도 화면에서 파일을 고르는 UI가 있으면 크기·확장자 제한은 필요합니다.)

### 사용 방법

**두 가지 방식**을 지원합니다. 편한 방식을 선택하세요.

#### 방식 A: 대화형 (기본, 추천)

> 템플릿을 직접 채울 필요 없이, Claude가 섹션 순서대로 질문하고 답변을 받아 채워나갑니다.

```
1. 이 템플릿 파일을 Claude에게 전달
2. "기능명세서를 작성하고 싶어" 또는 "이 템플릿으로 기능명세서를 생성해줘" 라고 요청
3. Claude가 §1 기능 개요부터 순서대로 질문 → 답변하면 다음 섹션으로 진행
4. 모든 섹션 완료 후 기능명세서 초안 생성
5. 검토 → 피드백 → 최종본 확정
```

**Claude 진행 규칙**:
- **다른 정보의 md 파일을 읽을 때는 사용자의 확인을 받는다** (템플릿이 직접 참조하는 파일 제외 — 예: `.claude/skills/feature-spec/output_format.md`)
- 섹션 순서대로 질문하되, 한 번에 너무 많은 질문을 하지 않는다 (최대 2~3개)
- 답변이 충분하면 다음 섹션으로 넘어간다
- 답변이 부족하면 `💬 Claude 질문` 블록의 질문으로 보완한다
- DB 접속정보를 받으면 직접 접속하여 스키마를 조회하고 결과를 보여준다
- 각 섹션이 채워질 때마다 진행 상황을 요약해준다

#### 방식 B: 직접 작성형

> 작성자가 템플릿을 미리 채워서 Claude에게 전달합니다. 빈 섹션은 Claude가 대화로 보완합니다.

```
1. 이 템플릿을 복사하여 새 파일 생성 (예: `docs/<현재브랜치>/`input_보험료산출.md)
2. 아는 항목을 채워 작성
3. 작성한 파일을 Claude에게 전달하며 "이 템플릿으로 기능명세서를 생성해줘" 라고 요청
4. Claude가 채워진 항목을 확인하고, 비어있는 필수 항목은 대화로 질문
5. 기능명세서 초안 생성 → 검토 → 최종본 확정
```

---

### 작성 가이드

> 방식 B로 직접 작성하거나, 방식 A 도중 미리 채워두고 싶을 때 참고하세요.

- `(필수)` 항목은 반드시 작성 — 이게 없으면 명세서 생성 불가
- `(선택)` 항목은 아는 만큼 — 비워두면 Claude가 대화로 보완합니다
- 각 섹션의 `(예: ...)` 는 작성 참고용 예시입니다. **예시를 지우고 실제 내용으로 교체**하세요. 해당 없으면 예시를 지우고 비워두면 됩니다
- `💬 Claude 질문` 블록은 참고용 — 해당 섹션이 부족할 때 Claude가 물어볼 예상 질문입니다

## 3. 화면 레이아웃 [스코프: `full` · `frontend-only` · `frontend-with-mock`]

> **`backend-only` 스코프는 이 섹션 전체를 생략합니다.**
>
> 아래 단계로 작성하세요. **3-1은 필수**, 나머지는 선택(일부는 스코프별 필수)입니다.
> Claude가 이 정보를 조합하여 상세 레이아웃과 ASCII 와이어프레임을 생성합니다.

### 3-1. Step 1 — 레이아웃 패턴 선택 (필수)

> 아래에서 가장 가까운 패턴 번호를 선택하세요. 여러 개 조합도 가능합니다.

| 패턴 | 구조 | 설명 |
|---|---|---|
| **① 좌우 분할** | `[패널A \| 패널B]` | 왼쪽 트리/목록 + 오른쪽 상세/편집 (예: 상품기초정보) |
| **② 상하 분할** | `[검색/필터]` + `[데이터 테이블]` | 위에 조건, 아래에 결과 테이블 (예: 목록 조회 화면) |
| **③ 단일 테이블** | `[필터 + 그리드]` | 하나의 데이터 그리드에 인라인 필터 (예: 단순 CRUD) |
| **④ 탭 기반** | `[탭A \| 탭B \| 탭C]` + `[내용]` | 탭 전환으로 영역 구분 (예: 설정 화면) |
| **⑤ 마스터-디테일** | `[목록]` → 클릭 → `[상세 팝업/서브화면]` | 목록에서 항목 선택 시 상세 화면 전환 |
| **⑥ 대시보드** | `[카드1][카드2][카드3]...` | 여러 위젯/카드를 격자 배치 (예: 현황 대시보드) |
| **⑦ 기타** | 직접 설명 | 위 패턴에 해당하지 않는 경우 |

**선택**: (예: ①, ①+④ 조합)

### 3-2. Step 2 — 글로 보충 설명 (선택)

> 선택한 패턴에서 추가로 설명할 내용이 있으면 자유롭게 적어주세요.

```
(예: 
"① 좌우 분할인데, 왼쪽은 폴더/상품/담보 트리, 오른쪽은 상품 데이터 그리드.
가운데 드래그로 크기 조절. 왼쪽 접기 가능. 상단에 공통 헤더 있음.")
```

### 3-3. Step 3 — 참고 화면 (선택)

> 참고할 기존 화면이 있으면 스크린샷 경로나 링크를 적어주세요. 없으면 비워두세요.

```
(예:
- 기존 레거시 화면 캡처: /docs/legacy_screenshot.png
- 유사 기능 화면: "admin-portal의 사용자 관리 화면과 비슷"
)
```

### 3-6. 사용자 시나리오 (필수 · 프론트 포함 스코프)

> 주요 화면 동작별로 **정상 / 에지 / 거부** 케이스를 표로 적어주세요. 이 표는 feature-test 스킬이 E2E 시나리오 생성 시 참고하며, 옵션 C 구현의 화면 확인 게이트 기준이 됩니다.

**예시**:

| 구분 | 시나리오 | 사용자 액션 | 기대 화면 반응 |
|---|---|---|---|
| 정상 | 목록 로드 | 페이지 진입 | 그리드에 N건 표시 + 상단 건수 `총 N건` |
| 정상 | 검색 | 검색창에 키워드 입력 후 Enter | 일치 항목만 표시, 0건이면 빈 상태 |
| 에지 | 긴 텍스트 | 셀에 200자 입력 | 말줄임 처리 + 툴팁으로 전체 노출 |
| 거부 | 필수값 누락 | 저장 버튼 클릭, 필드 공백 | 저장 실패 토스트 + 해당 필드에 오류 표시 |
| 거부 | 권한 없음 | 일반 사용자가 삭제 버튼 시도 | 버튼 비활성 + 호버 시 안내 |

# 부록: 기획자를 위한 용어 설명

| 용어 | 쉬운 설명 |
|---|---|
| **테이블** | 데이터를 저장하는 엑셀 시트와 비슷한 것. 행(row)과 열(column)로 구성 |
| **컬럼** | 테이블의 열. 엑셀의 A열, B열 같은 것 |
| **FK (외래키)** | 다른 테이블의 데이터를 참조하는 연결 고리 |
| **API** | 프론트엔드(화면)와 백엔드(서버) 사이의 통신 규약. "이 주소로 이런 데이터를 보내면 이런 응답이 온다" |
| **트랜잭션** | 여러 작업을 하나로 묶어서, 중간에 실패하면 전부 원래대로 되돌리는 것 |
| **Lazy Loading** | 데이터를 한번에 다 불러오지 않고, 필요할 때(예: 폴더 클릭 시) 그때그때 불러오는 방식 |
| **와이어프레임** | 화면의 대략적인 배치를 보여주는 뼈대 그림 |
| **UUID** | 전 세계에서 겹치지 않는 고유한 ID 값 (예: `A6618480165860...`) |
| **idempotent** | 같은 작업을 여러 번 해도 결과가 동일한 것 (예: 같은 상품을 두 번 추가해도 1건만 추가됨) |
| **ASCII 와이어프레임** | 텍스트 문자(─ │ ┌ ┐ 등)로 그린 화면 설계도. 별도 도구 없이 메모장에서 작성 가능 |
